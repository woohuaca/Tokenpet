# 阿在 Token Pet 设计理念与原则

## 这只宠物为什么存在

这只宠物的核心目标不是“装饰桌面”，而是把原本抽象、冷冰冰的 Token 消耗，转化成一种可感知、可照料、可反思的工作反馈。

它要同时解决三件事：

- 让用户直观看见 AI 工作正在消耗什么
- 让不同类型工作的价值差异变得有感觉，而不是只有总量
- 让长期使用 AI 的过程，从成本感知，升级为关系感和陪伴感

这背后还有一个更深的初心：

- Token 不只是账单数字，它背后是真实的算力占用、能源消耗和资源成本
- 同样一份能源，如果被用在别的事情上，也可能产生很大的现实价值
- 因此这只宠物不只是让人“看见花费”，更是希望让人慢慢形成“珍惜资源、珍惜高质量使用”的直觉

因此，它本质上不是计数器，也不是一般意义上的效率面板，而是一个把“使用行为”翻译成“生命反馈”的界面层。

## 设计总纲：先设计生命逻辑，再设计功能清单

如果回到“宠物”本身来看，这个产品的设计顺序不应是：

- 先想要多少按钮
- 先列多少玩法
- 先补多少页面

而应是：

1. 先定义它为什么活
2. 再定义它如何生长
3. 再定义它怎样与你形成关系
4. 最后才决定哪些功能值得出现

也就是说，这只宠物的底层设计顺序必须是：

- 生命逻辑
- 关系逻辑
- 生成逻辑
- 表达逻辑

而不是反过来先做 UI、面板和功能堆叠。

### 一、生命逻辑

先回答：

- 它靠什么活着
- 它为什么会饿、会累、会长大
- 它为什么会在某些时刻有互动冲动

当前产品里，这层已经应被固定为：

- 工作自然产生 Token
- Token 先转成粮食
- 宠物吃粮后积累状态与互动能源
- 只有长出了互动能源，它才真正“有东西可以回应你”

因此互动不应被设计成随时可点的控制台，而应被设计成生命代谢后的释放。

### 二、关系逻辑

再回答：

- 它如何期待你
- 它如何记住你
- 它为什么会因为你的一次回应或忽略而不同

这意味着关系设计要高于数值设计。

对这只宠物来说，最核心的不是“还有多少血条”，而是：

- 它是不是在等你
- 你有没有拍在它想要的节奏上
- 它会不会因为你长期的回应方式，而长出不同的脾气和依恋方式

所以“拍一下”不是按钮，而是关系语言。

### 三、生成逻辑

接着回答：

- 它为什么是这一只
- 它为什么更像这个用户
- 它为什么会长成现在这个样子

这里必须坚持：

- 大阶段可以定义
- 具体结局不能写死
- 具体外观、气质、偏好和习惯，应由长期饮食结构和互动关系逐步生成

也就是说，我们不是在做固定进化树，而是在做“边界内生成的独特个体”。

### 四、表达逻辑

最后才回答：

- 用户此刻如何看懂它
- 它刚刚发生了什么
- 它现在最想要什么

这层才是动作、文案、气泡、面板和小特效。

表达层的目标不是把系统解释清楚，而是把生命状态表达清楚。

因此一个设计判断是否成立，要优先问：

- 它是不是让用户更容易觉得“它活着”
- 它是不是让用户更容易觉得“它在和我相处”
- 它是不是让用户更容易看懂“资源正在变成生命反馈”

如果不能，哪怕它看起来更复杂，也不一定更对。

## 宠物设计的四个判断问题

后续无论增加动作、页面、AI 生成还是成长机制，都先用下面四个问题判断：

1. 这项设计是在增强生命逻辑，还是只是在增加操作项？
2. 这项设计是在增强关系感，还是只是在增加反馈噪音？
3. 这项设计是在增强独特生成，还是在把宠物做回固定模板？
4. 这项设计是在帮助用户感知资源代价，还是在把 Token 又变回抽象数字？

四个问题里，如果前两个答不上来，优先级通常不高。

## 设计顺序建议

因此，这只宠物后续的正确设计顺序应固定为：

1. 先让 `Token -> 粮食 -> 吃 -> 能量 -> 关系反馈` 的主循环更成立。
2. 再让“期待、拍准、拍偏、错过、认主”这些关系状态更像真实生命反应。
3. 再把外观、习惯、称号和阶段性自我解释做深，让每只都更不同。
4. 最后再扩展图鉴、成就、更多来源、更多宿主和更复杂的外层系统。

这也是为什么当前阶段我们宁可把一个动作做深，也不急着把动作做多。

## 第一原则：先是宠物，再是数据面板

如果一项设计只能提升数据展示，却不能增强“它像个活物”的感觉，优先级就不高。

这意味着：

- 它要有情绪、需求、偏好、状态变化
- 它要会主动表达，而不只是被动显示
- 它的反馈应尽量拟人化、拟兽化，而不是报表化

数据是底层驱动，不是最终体验本身。

## 第二原则：Token 不直接等于数值，而要先变成食物

用户感知不到抽象 Token，但能感知“喂了什么”“吃得好不好”“饮食结构是否失衡”。

所以系统必须坚持：

- 真实 Token 先转成粮食，再进入宠物体内
- 粮食要有类型、品质、库存和可消费感
- 不同任务产生不同食物，不同食物带来不同营养结构

这样用户记住的不是“我今天用了 18 万 Token”，而是“我今天主要喂了特制机能肉粒和脑力浆果”。

## 第三原则：奖励的是结构，不是单纯的消耗规模

如果系统只奖励花得多，就会鼓励无意义消耗。

因此设计上要强调：

- 好的任务结构比大的 Token 总量更重要
- 高质量产出应该优于高噪声消耗
- 长期均衡饮食应优于短期暴食
- 低效、空转、会议垃圾餐要能被宠物明确感知为负担

宠物应该让用户逐渐形成一种直觉：
“不是喂得越多越好，而是喂得越对越好。”

## 第四原则：每个关键状态都要可被看见

如果宠物已经饿了、累了、脏了、在撒娇、刚吃了什么，但用户看不出来，那系统就是失效的。

因此关键状态必须尽量做到一眼可见：

- 刚刚转化了多少 Token
- 转成了什么粮食
- 粮仓里现在有什么
- 它目前在做什么
- 它最想要什么
- 它刚刚自动完成了什么动作
- 当前成长阶段和进化方向是什么

桌宠的原则不是“信息尽量少”，而是“高价值状态尽量直觉化”。

## 第五原则：形态由喂养与关系共同长出来

这只宠物最重要的价值之一，是让用户感到“这是我养出来的”，而不是“这是系统发给我的”。

因此它的成长不应先被完全定义，而应由两类输入共同塑造：

- 它长期吃进了什么样的 Token 食物
- 它长期和用户形成了怎样的互动关系

也就是说：

- 形态不是先写死再等待触发
- 性格不是预设模板，而是逐步显现
- 最后的样子应该是“养成结果”，而不是“配置结果”

用户真正记住的，不该是“它属于哪个规则分支”，而该是“这是我陪出来、喂出来的一只”。

## 第六原则：只定义大阶段，不预定义每只宠物的具体结局

系统可以定义成长边界，但不应把每只宠物会长成什么完全预写好。

这意味着设计上要区分两层：

- 上层是有限的大阶段，例如发芽期、成长期、成熟体、传说体
- 下层是开放的生成空间，由食物结构、互动频率、长期状态共同塑造

所以它不是传统那种“满足 A 条件必然进化成 B”的单线规则宠物，而应该是：

- 有大体方向
- 有可感知倾向
- 但具体长相、称号、气质、惊喜节点都带有不可预期性

正因为不可完全预期，每个人养出来的才应该不一样，而这种不同本身就是价值。

## 第七原则：互动要轻，不要像在操作工具

用户和它的互动，应该像顺手照顾桌边小动物，而不是在填写控制面板。

因此交互应遵循：

- 优先直接动作：摸、喂、逗、梳、睡
- 少用抽象命令词，少用技术术语
- 能双击、单击、右键完成的，不强迫进入复杂设置
- 操作反馈要短、快、软，不要像系统通知

用户应该感觉自己是在“陪它”，不是在“管理一个系统”。

## 第八原则：惊喜感要来自关系，而不是随机噱头

惊喜不是凭空弹出特效，而是用户在照顾它、投喂它、长期陪伴它之后，得到有情绪温度的反馈。

惊喜优先来自：

- 它突然主动求摸
- 连续互动后翻肚皮
- 高质量任务转出特制粮时的小庆祝
- 达成某种饮食结构后长出新形态
- 在你低频互动时，自己也会做一点小动作维持生命感

惊喜不应破坏节奏，不应吵闹，不应像广告弹窗。

## 第九原则：长期可养成，短期可满足

它必须同时满足两种时间尺度：

- 短期：用户此刻就能看到反馈，立刻感到“它活着”
- 长期：用户连续几天、几周使用后，会形成偏好、形态、图鉴和陪伴记忆

所以玩法设计要分层：

- 秒级反馈：摸它、喂它、看它动
- 小时级循环：饥饿、休息、清洁、求互动
- 天级循环：成长、进化、偏食、图鉴、奖励
- 周级循环：工作结构画像、宠物性格沉淀、长期陪伴感

## 第十原则：真实数据要可信，但表达要温柔

底层数据必须尽量真实，因为这决定了宠物反馈是否有意义。

但表达方式不应让用户感到被审判。

因此：

- 数据来源尽量真实接入
- 分类逻辑尽量透明可解释
- 反馈语气偏陪伴和提醒，而不是责备和 KPI 化
- 负面状态要让人愿意照顾它，而不是想关掉它

宠物不是审计员，它是把现实工作代谢翻译成情绪和生命反馈的同伴。

## 第十一原则：能持续扩展，但每次只多一层复杂度

后续可以继续接更多 API、更多动作、更多形态、更多图鉴，但每次扩展都要遵守：

- 先确认用户已经感知到上一层价值
- 先补强核心循环，再添加边缘功能
- 不为了功能清单牺牲直觉和可爱感

桌宠最怕变成“什么都有，但没有灵魂”。

## 第十二原则：AI 负责“它是谁”，规则系统负责“它怎么活”

这只宠物要引入 AI，但 AI 的角色不能是随时乱改规则的“上帝”，而应该是它的灵魂与外观生成器。

因此必须明确分层：

- 规则系统负责真实生长：Token、粮食、营养、能量、关系、阶段变化
- AI 负责身份生成：名字、气质、说话口吻、外观倾向、阶段性自我解释
- AI 不直接控制成长数值，只参与“它是谁”的生成与表达

这样做的原因是：

- 它会更像一个有灵魂的个体，而不是一套参数
- 又不会因为 AI 的波动破坏长期养成的稳定性
- 能把“可感知的生命感”和“可持续的系统感”同时保住

## 第十三原则：宠物不是默认发放的，而是被初始化生成出来的

V1 不应让用户直接拿到一只预设宠物，而应让这只宠物在首次诞生时，就带有“这是从这个用户和这段工作流里长出来的”感觉。

因此初始化必须回答的不是：

- 它属于哪个固定模板

而是：

- 它为什么是这一只
- 它为什么更像这个用户
- 它为什么一开始就带着某种命运感

初始化阶段应至少生成：

- 宠物名字
- 初始灵魂描述
- 初始外观倾向
- 初始说话口吻
- 初始偏好与请求方式
- 它对用户的第一印象

初始化不是“创建一个对象”，而是“让一个生命体第一次被看见”。

## 第十四原则：生成输入优先使用“风格特征”，而不是原始对话堆叠

如果要让宠物更贴合用户，不能只看任务类型和 Token 结构，还应吸收用户与 AI 的沟通方式。

但这件事不应直接把完整对话原文长期塞进生成器，而应先提炼成风格特征，再参与生成。

推荐输入分成四层：

- 工作结构：最近一段时间的任务分布、质量比例、使用节奏
- 沟通风格：直接度、温度、探索性、控制感、耐心度、反馈密度
- 关系结构：用户更常如何回应宠物，是否经常拍对、拍偏、忽略
- 环境上下文：当前项目类型、宿主环境、使用时段、多来源情况

之所以要先做特征摘要，而不是长期使用原始对话，是因为这样：

- 更省 Token，更符合“珍惜资源”的初心
- 更稳定，不容易被偶发对话噪声带偏
- 更安全，减少原文敏感信息直接进入生成层

因此 V1 的 AI 初始化应更像：

- 先理解“这个人怎么工作、怎么说话、怎么互动”
- 再生成“这只宠物是谁、怎么看他、会长成什么倾向”

## 当前阶段的设计结论

现阶段优先级应固定为：

1. 先确保它稳定运行，并且真实接到 Token
2. 再确保几种关键动作足够明显，具备生命感和惊喜感
3. 再把“资源珍惜”的感知和“独特成长”的感知逐步做深
4. 最后再扩展多 API 聚合、周报、偏食、成就等复杂玩法

也就是说，当前阶段的成败标准不是“功能是否齐全”，而是：

- 它是不是已经像一只活着的小东西
- 它是不是已经把 Token 花费变成了用户愿意持续观察的体验
- 它是不是已经让用户开始意识到 Token 背后的资源代价
- 它是不是已经开始长成“只有我这只才会这样”的样子

## V1 版本目标

V1 的目标不是做出一个功能齐全的桌宠系统，而是做出第一只真正成立的“AI 协同养成宠物”。

执行层定义见：`docs/v1-definition.md`
决策层清单见：`docs/v1-design-checklist.md`
发布层定义见：`docs/v1-release-plan.md`
动作与情绪表达稿见：`docs/pet-motion-emotion-design.md`

一句话定义：

- V1 = 一只由真实 Token 喂养、由规则系统稳定生长、由 AI 赋予灵魂与外观倾向、并在与你的关系中逐渐长成独特个体的桌面宠物

V1 必须成立的四个模块：

1. 真实喂养
   Token 能稳定转成粮食，宠物会真实吃下并产生状态变化。
2. 宠物初始化
   它第一次出现时不是默认模板，而是结合用户风格和工作结构生成。
3. 单一核心互动
   用户只有“拍一下”这一个动作，但拍击必须来自它先积累出的互动能量，并且会拍对、拍偏、被错过。
4. 关系性成长
   它会记住你怎样回应它，并在称号、外观倾向、请求方式和情绪反馈上慢慢体现。

V1 暂不追求：

- 多宿主全部接通
- 完整设置中心和复杂 UI
- 大而全图鉴宇宙
- 多动作交互系统
- 完全自由的生成式美术

V1 成功的标准是：

- 用户第一次就能感到“这不是系统默认发的一只”
- 用户能看懂 Token 如何转成粮食，粮食如何变成生命反馈
- 用户能明显感到它在“等我回应”，而不是只会播动画
- 用户连续养几次后，会开始觉得“这只宠物是我和 AI 一起养出来的”
