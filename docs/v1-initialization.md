# 阿在 Token Pet V1 初始化方案

## 初始化结论

V1 的初始化不是“创建默认宠物”，而是完成一次受约束的“诞生生成”。

初始化的目标是让用户第一次看到它时，就能感到：

- 这只不是系统默认发的一只
- 这只和我的工作方式、沟通方式有关
- 这只从一开始就带着某种灵魂和外观倾向

## 初始化分层

V1 初始化采用两层结构：

1. 风格摘要层
   从工作流和对话中提炼稳定特征，而不是直接长期使用原始对话。
2. 身份生成层
   根据这些特征生成宠物名字、灵魂、说话口吻、外观倾向和第一印象。

## 输入

V1 初始化使用四类输入：

### 一、工作结构

- 最近一段时间的任务类型分布
- 最近一段时间的 Token 消耗结构
- 最近的质量信号
- 最近的工作节奏

### 二、沟通风格

从近期对话中提炼：

- 直接度
- 温和度
- 探索性
- 控制感
- 耐心度
- 协作度

V1 原则：

- 使用风格特征摘要
- 不长期堆叠原始对话内容

### 三、环境上下文

- 当前宿主
- 当前项目环境
- 当前宠物已有状态

### 四、固定 seed

- 保证同一批输入能生成相对稳定的一只
- 避免每次启动都重新长成另一只

## 输出

V1 初始化至少输出：

- 名字
- 灵魂描述
- 说话口吻
- 外观倾向
- 对用户的第一印象
- 出生描述
- 风格向量

这些输出中：

- 规则系统负责持久化和后续演化
- AI 或生成器负责第一次赋形

## 当前实现策略

当前版本先采用本地生成器作为 V1 初始化落地方式。

原因：

- 可以先把“初始化存在”这件事落进产品
- 可以先验证输入结构和输出格式是否成立
- 后续替换成真正的 AI 生成器时，不必推翻状态结构

也就是说，当前版本已经有：

- 初始化输入结构
- 风格摘要字段
- 名字、灵魂、外观倾向、口吻、第一印象、出生描述这些输出槽位

后续如果要切到真正 AI 生成，只需要替换“身份生成器”，不需要重写宠物主循环。

## V1 发布前必须保证

1. 初始化只在首次诞生或缺失身份信息时触发。
2. 初始化结果生成后会稳定持久化。
3. 主界面能看见名字，图鉴能看见完整身份信息。
4. 初始化生成结果不会频繁跳变。
5. 后续成长是在初始化基础上继续长，而不是反复重生。

## V1 发布后再增强

这些放到后续版本：

- 真正接入 AI 模型做初始化生成
- 加入用户一句愿望输入作为显式初始化条件
- 让外观倾向更深入驱动绘制参数
- 在关键阶段重新生成“自我解释”
