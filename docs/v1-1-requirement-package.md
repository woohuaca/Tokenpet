# 阿在 Token Pet `v1.1.x` 需求包

## 版本结论

`v1.1.x` 的目标不是加更多功能，而是把当前已经成立的主循环，推进到第一次使用就更像一个生命体。

这一版从属于两条并进主线：

- 生命代谢线：
  `Token 统计 -> 分类 -> 转化为粮食与能量 -> 代谢与消耗 -> 状态平衡 -> 生长与养成`
- 情绪价值线：
  `第一印象 -> 期待感 -> 回应感 -> 被记住感 -> 陪伴感 -> 个体关系感`

这一版只围绕五个模块展开：

1. 首次诞生体验
2. 认主完成反馈
3. 进食、等待、拍准、拍偏、错过差异与互动特效
4. 状态文案与轻提示
5. 首次上手说明

一句话说，`v1.1.x` 要解决的是：

- 现在它已经能跑，但还不够让用户第一次就被打动。

## 版本要回答的问题

`v1.1.x` 需要同时回答三个问题：

1. 用户第一次看到它时，会不会觉得“这不是系统默认的一只”？
2. 用户第一次轻拍它时，会不会明确感觉到“关系真的开始了”？
3. 用户在之后几轮使用里，会不会开始主动理解它现在的期待？

如果这三个问题答不好，`v1.1.x` 就不算完成。

## 版本边界

### 这一版必须做

- 强化第一次见面的情绪成立感
- 强化认主完成这一刻的关系成立感
- 拉开关键情绪时刻的反馈差异
- 让状态表达更像生命提示，而不是状态播报
- 让新用户更快理解“怎么开始养它”
- 不破坏底层 `Token -> 能量 -> 养成` 的解释性和一致性
- 明确推进情绪价值线里的第一印象、期待感和回应感

### 这一版明确不做

- 不增加大量新动作面板
- 不新增复杂养成系统
- 不扩多宿主与多系统
- 不做完整图鉴宇宙
- 不把 AI 引入每一次实时互动

## 模块一：首次诞生体验

### 要解决的问题

当前初始化在结构上已经成立，但第一次见面更像“已经生成完毕”，还不够像“它刚刚在你面前出生”。

### 目标状态

用户第一次看到它时，要明确感到三件事：

- 它是刚刚诞生的
- 它和我的工作方式有关
- 它从一开始就带着某种个体气质

### 这版必须补齐

- 首次诞生时的欢迎演出
- 名字、第一印象、出生描述的出场顺序
- 更明确的“这只已经属于当前用户上下文”的确认感
- 初次诞生后的第一句引导话术

### 设计要求

- 演出要短，不拖沓
- 信息要少，但必须准
- 不做长篇设定说明
- 重点不是讲世界观，而是讲“这只是谁”

### 验收标准

- 新用户第一次见到时，能清楚记住这只的名字或气质
- 新用户能说出“它像是被生成出来的”，而不是“系统给了一只默认宠物”
- 首次诞生流程不会打断启动成功率

## 模块二：认主完成反馈

### 要解决的问题

现在认主逻辑已经存在，但“完成认主”这一下还不够像一个关系节点，更像一轮正确交互。

### 目标状态

第一次拍对并完成认主时，用户要明确感觉到：

- 这不是普通拍击
- 它已经真正认得我
- 从这一刻开始，之后的互动会不一样

### 这版必须补齐

- 认主成功的专属动作或姿态
- 认主成功的专属文案
- 认主完成后的短余波反馈
- 认主前后状态差异的可见化

### 设计要求

- 认主时刻必须和普通拍准区分开
- 反馈要有完成感，但不能太吵
- 认主完成后，后续等待姿态要有细微变化，证明关系状态已经改变

### 验收标准

- 用户第一次完成认主后，能明确说出“刚刚那一下和普通反馈不一样”
- 用户能理解认主不是自动发放，而是通过互动建立的
- 认主后再看宠物，会感到它和刚出现时已经不一样

## 模块三：进食、等待、拍准、拍偏、错过差异与互动特效

### 要解决的问题

当前这些状态已经有差异，但强弱层级还不够稳定，用户未必总能直觉分清“它现在是在满足、别扭、失落，还是只是普通待机”。

### 目标状态

用户即使不细读文案，也能从动作和轻特效大致判断当前时刻属于哪一类：

- 进食
- 等待
- 拍准
- 拍偏
- 错过

### 这版必须补齐

- 进食反馈更像“真的在吃到东西”
- 等待反馈更像“它正在等你”
- 拍准反馈更像“被正确接住”
- 拍偏反馈更像“接住了，但没完全对”
- 错过反馈更像“期待落空”

### 互动特效要求

- 特效要服务于情绪差异，不是为了更热闹
- 单次轻、短、准
- 视觉上尽量用少量但明确的变化
- 不做高频抢眼动画

### 最低区分要求

- 进食和等待不能混成同一种“普通动一下”
- 拍准和拍偏不能只是文案不同
- 错过必须有明显收回感，而不是直接回到待机

### 验收标准

- 用户在不看说明时，能大致分清这五类反馈
- 用户会开始主动判断“刚刚是不是拍准了”
- 动作和特效增强情绪表达，但不会让桌面打扰度明显上升

## 模块四：状态文案与轻提示

### 要解决的问题

当前状态表达能传信息，但有时更像系统播报，不够像生命体在释放自己的状态和期待。

### 目标状态

状态文案和轻提示要同时做到：

- 用户看得懂
- 它说得像自己
- 不打扰

### 这版必须补齐

- 把状态播报改成更偏生命表达的短句
- 把“它现在想要什么”说得更直接但不命令式
- 让轻提示更能区分进食后、等待中、拍偏后、错过后的状态
- 让提示频率更克制，减少无效重复

### 文案原则

- 一次只传一个重点
- 先传情绪，再传状态
- 不堆术语
- 不用过多解释性句子

### 验收标准

- 用户能从文案里更快理解它当前期待
- 用户不会觉得它像在弹系统 toast
- 同一状态下的文案有稳定语气，不显得人格跳变

## 模块五：首次上手说明

### 要解决的问题

当前上手说明已经能把人带起来，但还偏“说明书”，不够像把用户温柔地引进一段陪伴关系。

### 目标状态

新用户在最短时间内理解四件事：

1. 怎么把它跑起来
2. 它吃的是真实 Token
3. 先看它吃，再轻拍一次
4. 拍准、拍偏、错过会不一样

### 这版必须补齐

- 更短的首次上手说明
- 更明确的第一轮建议动作
- 更清楚的“不要一上来就重置或乱点”的边界提示
- README 和上手卡之间的角色分工更清楚

### 说明结构要求

- 先讲三步怎么开始
- 再讲你会看到什么
- 最后讲常用操作和边界

### 验收标准

- 新用户不需要额外口头解释，也能完成首次启动和第一次互动
- 新用户能在 1 分钟内理解这不是普通桌宠，而是 Token 驱动的桌面生命体
- 上手说明不会压过产品本体的第一印象

## 五个模块的优先级

### P0

- 首次诞生体验
- 认主完成反馈
- 进食、等待、拍准、拍偏、错过差异

### P1

- 状态文案与轻提示
- 首次上手说明

原因不是后两项不重要，而是前 3 项更直接决定情绪价值线能不能先成立。

## 版本交付清单

`v1.1.x` 交付时，建议至少包含：

- 一轮新的首次诞生流程
- 一套认主完成反馈
- 一组关键情绪动作与特效差异
- 一版新的状态短文案
- 一版新的首次上手卡或 README 上手入口

## 版本成功标准

如果 `v1.1.x` 做完，外部反馈里应该更容易出现这些话：

- “它真的像刚刚出生出来的”
- “第一次拍到它的时候有点心动”
- “我开始会看它现在到底想要什么”
- “我能感觉出来刚刚是拍准了还是拍偏了”

如果用户仍然主要反馈：

- “机制有了，但没什么感觉”
- “像个会动的小挂件”

那说明 `v1.1.x` 还没有真正完成。

另外，如果这一版虽然更好看了，但用户开始更看不懂：

- Token 是怎么被统计的
- 为什么会转成这种粮
- 为什么它现在是这种状态

那也说明 `v1.1.x` 偏离了正确方向。

同样，如果这一版虽然逻辑更完整了，但用户仍然感受不到：

- 它在等我
- 它记得我
- 我刚刚真的回应到了它

那说明情绪价值线也没有被真正推进。

## 当前判断

`v1.1.x` 不应该被做成一个“补 UI 和补文案”的版本，而应该被做成：

- 阿在 Token Pet 第一次真正让人心里一软的版本。
