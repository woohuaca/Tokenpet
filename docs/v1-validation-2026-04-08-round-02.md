# 阿在 Token Pet V1 第二轮验证记录

## 本轮目的

这轮不是全量重验，而是专门针对首轮记录里的关键 `P1` 做补充验证：

- 启动
- 关闭
- 再次启动
- 单实例链路

目标是回答：

- 当前 `launch_desktop_pet.command` 是否真的具备单实例能力
- 当前 `stop_desktop_pet.command` 是否真的能把桌宠和 bridge 一起停掉

## 本轮结论

- 总结论：`通过，但仍需回归观察`
- 当前最新判断：
  - 双实例问题在本轮最终复测里未再出现
  - `desktop-pet.pid` 已能与真实常驻桌宠 PID 对齐
  - bridge 改为由桌宠本体自愈拉起后，`codex-bridge.pid` 也已能与真实 bridge PID 对齐
  - 启动脚本的职责已收窄为“拉起桌宠”，不再和桌宠本体争抢 bridge 守护权

这意味着：

- `V1` 的“单实例稳定”在当前版本下可以先打勾
- 但发布前仍建议保留一轮回归观察，确认多次日常启停下不会回退

## 本轮真实观察

### 1. 关闭脚本的第一次验证

观察结果：

- 执行 `scripts/stop_desktop_pet.command` 后，bridge 仍然残留
- 早期版本脚本中的 `ps` 调用在当前验证环境里报了 `operation not permitted`

结论：

- 原脚本对 bridge 和 pet 的停止能力不可靠

### 2. 启动后重复启动验证

观察结果：

- 当桌宠处于单实例状态时，再次执行 `scripts/launch_desktop_pet.command`
- 桌宠 PID 从 `42350` 变成 `42402`

结论：

- 启动脚本在“已有单实例且状态干净”时，具备替换旧实例的能力

### 3. 停止脚本修订后再测

本轮动作：

- 把停止脚本从依赖 `ps` 改成按路径 `pkill -f` 兜底
- 再次执行关闭链路

观察结果：

- 脚本本身不再报错
- 但桌宠进程仍未真正退出

结论：

- 问题不只是匹配命令本身
- 当前桌宠停止链路仍未真正成立

### 4. 单实例 PID 文件链路排查

观察结果：

- 代码中存在 `claimSingleInstance()` 与 `releaseSingleInstance()`
- 按实现应写出 `runtime/desktop-pet.pid`
- 在本轮早期排查时，工作区内一度未找到 `desktop-pet.pid`
- 但后续 fresh launch 后，`desktop-pet.pid` 又恢复落盘

结论：

- 当前单实例逻辑并非完全失效，而是在某些启停顺序下会进入不一致状态
- 这会直接影响停止脚本和重复启动时的稳定性

### 4.1 bridge 启动链补充观察

观察结果：

- fresh launch 后，`codex-bridge.pid` 会被写出
- 但进程列表中不一定存在对应 bridge 进程
- 手动直接运行 `python3 -u scripts/codex_usage_bridge.py` 时，bridge 可以稳定常驻

结论：

- bridge 本体并非不能运行
- 当前问题更像是“启动脚本如何守护 bridge”而不是 bridge 自身逻辑崩溃

### 5. 严格顺序验证的结果

观察结果：

- 在“旧实例没有真正停干净”的情况下再次启动
- 当前验证中出现了两只桌宠实例同时存在的情况

结论：

- 当前版本仍不能把“单实例稳定”视为已通过

### 6. 启动链收紧后的补充复测

本轮新增动作：

- 启动脚本改为直接后台拉起桌宠，不再依赖 `open`
- 启动前显式清理旧实例，并等待旧进程退出
- 启动后优先等待桌宠本体自行写入 PID，再回退到进程表补写
- 应用内 `claimSingleInstance()` 也增加了 `SIGTERM -> SIGKILL` 的双阶段清理

观察结果：

- 重复启动时，当前没有再稳定复现“两只桌宠同时常驻”的情况
- 但 `desktop-pet.pid` 仍可能写到非最终常驻 PID
- `ps` 可见的桌宠 PID 与 `desktop-pet.pid` 在当前环境里仍可能不一致
- 这意味着：双实例风险已经被压下去一大半，但 PID 真值判断仍未完全可靠

结论：

- “重复启动容易叠双实例”这个问题已明显改善
- 但“PID 链路可信”这一条还不能视为完全通过

### 7. 桌宠接管 bridge 守护后的最终复测

本轮新增动作：

- 启动脚本改为只拉起桌宠本体，bridge 不再由脚本后台守护
- 桌宠本体在启动时主动 `ensureBridgeRunning()`
- 桌宠在后台 tick 中也会补一次 bridge 自愈
- 桌宠本体启动方式改为优先 `open`，失败时再回退直接执行

观察结果：

- fresh launch 后，真实进程列表稳定出现 `1` 只桌宠 + `1` 条 bridge
- `desktop-pet.pid` 与真实桌宠 PID 一致
- `codex-bridge.pid` 与真实 bridge PID 一致
- 再次执行 `launch_desktop_pet.command` 后，真实进程列表仍保持 `1` 只桌宠 + `1` 条 bridge
- 二次拉起后，两个 PID 文件仍与真实常驻进程一致

结论：

- 当前版本已经通过“重复启动后仍保持单实例”的专项验证
- 当前阻塞点已从“单实例不稳”收敛为“上线前继续做一轮回归观察”

## 当前判断

结合第一轮与第二轮结果，`V1` 当前状态更准确的描述应是：

- 主循环已经基本成立
- 关系反馈已经很完整
- 发布配套文档已经基本够用
- 但启动 / 停止 / 单实例链路仍然是最后一个关键风险点
