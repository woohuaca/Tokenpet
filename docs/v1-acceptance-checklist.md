# 阿在 Token Pet V1 验收清单

## 这份清单的作用

这份清单不是设计原则，也不是发布策略，而是 `V1` 真正准备发出 `dev preview` 之前要逐项打勾的验收表。

它要回答的是：

- 现在到底能不能发
- 还差的是体验问题、稳定性问题，还是发布配套问题
- 私测前必须人工走完哪些链路

使用原则：

- 一项不过，就不应把 `V1` 视为“可发布”
- 可以有“已完成 / 待验证 / 未完成”三种状态
- 这份清单优先服务 `Codex + macOS companion` 首发路径

## 一、安装与启动验收

### 1. 最小启动路径可用

必须满足：

- 运行 `scripts/launch_desktop_pet.command` 能成功拉起桌宠
- 首次缺少二进制时，能自动编译
- 二次启动时，不要求用户手动清理旧实例

通过标准：

- 新机器按 README 步骤能跑起来
- 桌宠启动后，桌面上只出现一只

### 2. 关闭与重置路径可用

必须满足：

- `scripts/stop_desktop_pet.command` 能关闭桌宠和 bridge
- `scripts/reset_pet_data.command` 能停止后清空运行态
- 重置后重新启动，能生成一只新的宠物

通过标准：

- 关闭后不残留桌宠进程
- 重置后不沿用旧宠物身份和关系

### 3. 自启路径可用

必须满足：

- `scripts/install_autostart.command` 可安装
- `scripts/uninstall_autostart.command` 可移除

通过标准：

- 路径和脚本在本机有效
- 不会因为自启重复拉起多只桌宠

## 二、首次诞生验收

### 4. 第一眼不是默认宠物

必须满足：

- 第一次启动时会生成名字
- 会生成灵魂描述、外观倾向、说话口吻、第一印象
- 用户能感到“这是从我这里长出来的一只”

通过标准：

- 不是匿名默认体
- 图鉴和主面板都能看见身份信息

### 5. 认主第一下成立

必须满足：

- 初见阶段会明确提示先完成第一次轻拍
- 第一次轻拍后会完成认主
- 认主完成后会有专属反馈，不与普通拍击混淆

通过标准：

- `firstBondAt` 正确写入
- 认主后请求与状态切换正常
- 用户能明显感到“它认住我了”

## 三、真实喂养验收

### 6. Codex Token 真正进粮仓

必须满足：

- 能从 Codex 会话读取真实 Token 事件
- Token 先转成粮食，而不是直接加经验
- 最近一笔转粮在桌宠上可见

通过标准：

- `codex-feed.json` 有新增事件
- `pet-state.json` 的粮仓和最近转粮字段正确变化

### 7. 自动吃粮链路成立

必须满足：

- 粮仓里有粮时，宠物会按状态自动吃
- 吃粮后状态、活动、最近动态会变化
- 不会出现一口把所有库存吃光的异常

通过标准：

- 能看到“转粮 -> 存粮 -> 自动吃”的完整链路
- 吃完后互动能量能增长

## 四、互动与关系验收

### 8. “拍一下”足够成立

必须满足：

- 用户只有一个核心动作：拍一下
- 拍击由互动能量决定，不是随时可用
- 至少分轻拍、中拍、重拍三档

通过标准：

- 没攒够能量时，拍不出正式反馈
- 能量不同，按钮和反馈强度不同

### 9. 拍准、拍偏、错过有明显差异

必须满足：

- 拍准时有明显正向反馈
- 拍偏时有明显“还没拍在点上”的反馈
- 错过和忽略时有明显等待或失落反馈

通过标准：

- 不看代码，仅从动作、气泡、文案也能区分
- 至少能清楚感到三者不是同一种状态

### 10. 关系会留下长期痕迹

必须满足：

- 拍击历史会影响称号、习惯、期待方式和外观倾向
- 同一只宠物连续互动后会显出关系差异
- 重启后关系不会丢

通过标准：

- 存档和重启后，关系字段仍保留
- 图鉴或主面板能看出“这只被怎样养过”

## 五、独特成长验收

### 11. 不是固定模板宠物

必须满足：

- 允许定义大阶段
- 但外观、习惯、偏好和称号不应完全写死
- 长期饮食结构和关系会参与生成

通过标准：

- 两种明显不同的养法，能产生可感知差异
- 差异不只停留在数值条

### 12. AI 角色边界正确

必须满足：

- AI 负责“它是谁”
- 规则系统负责“它怎么活”
- 不把每一次互动都做成高成本 AI 调用

通过标准：

- 初始化和阶段性解释有 AI 角色定位
- 主循环不依赖实时 AI 才能成立

## 六、稳定性验收

### 13. 单实例稳定

必须满足：

- 重复启动不会出现多只桌宠
- 关闭再启动后状态正确恢复

通过标准：

- 进程检查时只保留一只桌宠和一条 bridge

### 14. 状态持久化稳定

必须满足：

- `pet-state.json` 长期可读
- 新字段加入后旧存档能迁移
- 重启后宠物身份、关系、粮仓不异常跳变

通过标准：

- 旧宠物不会被误判成未认主
- 存档读取失败时至少能回到默认安全状态

### 15. bridge 运行稳定

必须满足：

- `codex_usage_bridge.py` 持续运行
- 中断后可恢复
- 不重复消费同一事件

通过标准：

- 连续一段真实使用后，事件 id 不重复回放

## 七、发布配套验收

### 16. 私测文档够用

必须满足：

- README 能覆盖安装、启动、关闭、重置、隐私
- 私测用户能直接看上手卡跑起来

通过标准：

- 至少存在：
  - `README.md`
  - `docs/v1-private-beta-quickstart.md`
  - `docs/v1-release-plan.md`

### 17. 已知限制说明清楚

必须满足：

- 当前首发范围讲清楚
- 不稳定边界讲清楚
- 不要让私测用户误以为是正式版

通过标准：

- 文档里明确出现 `developer preview`
- 说明当前以 `Codex + macOS` 为主

## 八、最终发布判断

只有当下面三层同时成立，`V1` 才可以被视为“可发私测”：

1. 主循环成立
   `Token -> 粮食 -> 吃 -> 能量 -> 拍 -> 成长` 没有断链。
2. 生命感成立
   用户能明显感到它会等待、会回应、会记住自己。
3. 发布配套成立
   用户能装、能关、能重置、能理解数据边界。

如果只成立前两层，而第三层不成立，它仍然只是“可玩的原型”，不是“可发的私测版”。

## 当前建议的验收顺序

正式验收时，建议按下面顺序走：

1. 启动
2. 首次诞生
3. 真实转粮
4. 自动吃粮
5. 认主首拍
6. 拍准 / 拍偏 / 错过
7. 关闭与重启恢复
8. 重置生成新宠物
9. 文档和隐私说明复核
