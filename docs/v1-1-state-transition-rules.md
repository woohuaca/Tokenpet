# 阿在 Token Pet `v1.1.x` 第一批状态交互规则表

## 结论

这份规则表服务于 `v1.1.x` 第一批 7 个关键状态：

1. 出生
2. 认主
3. 进食
4. 等待
5. 拍准
6. 拍偏
7. 错过

它回答的不是“状态看起来怎样”，而是：

- 什么时候进入
- 什么时候退出
- 持续多久
- 状态冲突时谁优先
- 哪些状态可以被打断，哪些不能

有了这张表，动作、特效、文案和代码状态机才容易统一。

## 使用规则

每个状态统一按下面 6 项定义：

1. 进入条件
2. 退出条件
3. 建议持续时长
4. 可否被打断
5. 优先级
6. 冲突规则

## 总体优先级顺序

建议第一批状态统一按下面顺序处理优先级：

1. `出生`
2. `认主`
3. `拍准`
4. `拍偏`
5. `错过`
6. `进食`
7. `等待`

解释：

- `出生` 和 `认主` 都是关系节点，应优先于普通状态
- `拍准 / 拍偏 / 错过` 是即时互动结果，应优先于背景态
- `进食` 是生命代谢反馈，应高于普通等待
- `等待` 是基础持续态，应作为默认可回落状态

## 一、出生

### 进入条件

- 首次生成宠物
- 或重置后生成新宠物

### 退出条件

- 出生文案与姿态展示完成
- 或用户已看到首次引导

### 建议持续时长

- `2.5s - 4.0s`

### 可否被打断

- 默认不可被普通等待打断
- 可被认主前必要的系统初始化完成事件推进结束

### 优先级

- 最高

### 冲突规则

- 出生期间不进入普通等待
- 出生期间即使有进食，也优先先完成出生展示，再平滑切到进食或等待

## 二、认主

### 进入条件

- 首次达成认主判定

### 退出条件

- 认主专属动作与提示播放完成
- 认主余波结束

### 建议持续时长

- `1.8s - 3.0s`

### 可否被打断

- 不可被等待和进食打断
- 原则上不应被普通拍准覆盖

### 优先级

- 仅次于出生

### 冲突规则

- 如果认主与拍准同时成立，只进入认主
- 认主结束后回到“认主后等待”或“进食后状态”，而不是普通待机

## 三、进食

### 进入条件

- 一次有效进食结算完成

### 退出条件

- 进食动作与轻特效结束
- 状态更新完成

### 建议持续时长

- `1.0s - 1.8s`

### 可否被打断

- 可被认主打断
- 可被即时互动结果打断

### 优先级

- 高于等待，低于互动结果

### 冲突规则

- 如果进食与等待连续发生，先显示进食，再回到等待
- 如果进食后立即触发拍击结果，优先展示拍准 / 拍偏 / 认主

## 四、等待

### 进入条件

- 已有互动期待
- 当前没有更高优先级状态正在播放

### 退出条件

- 用户做出回应
- 等待窗口超时
- 被更高优先级状态覆盖

### 建议持续时长

- 持续态，不设固定上限

### 可否被打断

- 可被任何更高优先级状态打断

### 优先级

- 最低持续态

### 冲突规则

- 等待永远作为默认回落状态
- 不主动覆盖任何一次性结果状态

## 五、拍准

### 进入条件

- 当前回应与期待匹配
- 但不属于首次认主

### 退出条件

- 拍准动作与短余波结束

### 建议持续时长

- `1.0s - 1.6s`

### 可否被打断

- 不可被等待打断
- 可被认主覆盖

### 优先级

- 高于拍偏、错过、进食、等待

### 冲突规则

- 与认主同时命中时，由认主接管
- 拍准结束后，如果仍有期待窗口，则回到等待；否则回到普通状态

## 六、拍偏

### 进入条件

- 用户做出回应
- 但力度或时机未命中当前期待

### 退出条件

- 拍偏短反馈结束

### 建议持续时长

- `0.9s - 1.4s`

### 可否被打断

- 不可被等待打断
- 可被认主或更高优先级节点覆盖

### 优先级

- 低于拍准，高于错过、进食、等待

### 冲突规则

- 拍偏一旦触发，当轮不再进入错过
- 拍偏结束后可回到等待或普通状态，视期待窗口是否还在

## 七、错过

### 进入条件

- 等待窗口结束
- 用户未给出正确回应

### 退出条件

- 错过收回动作结束

### 建议持续时长

- `1.0s - 1.8s`

### 可否被打断

- 原则上不被等待打断
- 可被新的更高优先级即时事件打断

### 优先级

- 低于拍准、拍偏，略高于进食与等待

### 冲突规则

- 错过只在“没有发生回应”的情况下触发
- 错过结束后不应立即高频重复进入第二次错过，需回到普通等待节奏

## 八、状态切换建议

建议第一批实现时，把状态切换统一约束为：

- `出生 -> 等待`
- `出生 -> 进食`
- `等待 -> 拍准`
- `等待 -> 拍偏`
- `等待 -> 错过`
- `等待 -> 认主`
- `进食 -> 等待`
- `拍准 -> 等待`
- `拍偏 -> 等待`
- `错过 -> 普通状态或等待`
- `认主 -> 认主后等待`

不建议第一批出现过多跨层跳转。

## 九、节奏限制

为了避免闪烁、抖动、抢状态，建议统一加三条限制：

1. 同一时刻只允许一个主状态。
2. 一次性结果状态结束前，不进入第二个同级结果状态。
3. 连续状态切换之间应有最小间隔，避免一帧内连跳。

建议最小间隔：

- `0.25s - 0.4s`

## 十、第一批最关键验证项

- 出生不会被普通等待抢掉
- 认主不会被普通拍准覆盖
- 拍偏不会和错过同时出现
- 进食不会打断认主和拍击结果
- 错过不会高频重复触发
- 等待始终是默认回落状态

## 当前判断

只要这张规则表成立，`v1.1.x` 第一批 7 个状态就不只是“看起来像”，而是已经具备一套真正可实现的状态机边界。
