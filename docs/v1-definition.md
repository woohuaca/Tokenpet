# 阿在 Token Pet V1 版本定义

## V1 结论

V1 的目标已经可以定下：

- V1 = 一只由真实 Token 喂养、由规则系统稳定生长、由 AI 赋予灵魂与外观倾向、并在与你的关系中逐渐长成独特个体的桌面宠物

这不是一个“功能尽量多”的版本，而是第一个必须让用户明确感到“它活着，而且这只就是我的”的版本。

## V1 要解决什么问题

V1 要同时解决四个问题：

1. Token 消耗太抽象，用户无法直觉感知自己正在消耗什么资源。
2. AI 使用行为虽然频繁，但很难沉淀成有情绪、有关系的长期体验。
3. 电子宠物如果只是固定规则分支，很快会被攻略化，失去独特性。
4. 如果没有 AI 参与“它是谁”的生成，这只宠物会像一套设定，而不像一个个体。

因此，V1 不是做一只会动的面板宠物，而是做一个把资源消耗、关系互动和 AI 生成整合起来的生命化界面。

## V1 核心承诺

用户第一次用它时，应该清楚感到三件事：

1. 这只宠物吃的是真实工作产生的 Token。
2. 这只宠物不是默认发放的，而是从我的工作方式和沟通方式里生成出来的。
3. 我之后怎么回应它，会继续改变它的性格、期待和样子。

## V1 核心循环

V1 的核心循环固定为：

1. 工作产生 Token
2. Token 转成不同粮食
3. 宠物吃掉粮食并积累状态与互动能量
4. 它长出明确的期待，告诉你现在想被怎样回应
5. 你用唯一动作“拍一下”回应它
6. 回应是否拍对、拍偏、错过，会影响它的情绪与下一步发育
7. 长期下来，宠物长成一只只有你这只才会这样的个体

这条循环是 V1 的主骨架，后续功能都不能破坏它。

## V1 范围

### 一、真实喂养

必须成立：

- 稳定读取真实 Token 来源
- Token 先转粮，再喂养，不直接变经验值
- 不同任务类型和质量生成不同粮食
- 粮仓、最近转粮和自动吃粮逻辑可被看见

当前状态：

- Codex 已接通
- 外部标准事件源已具备接入口

V1 要求：

- 先把 Codex 跑稳
- 外部来源保留标准接入口，但不要求在 V1 全部接完

### 二、宠物初始化

必须成立：

- 首次不是发默认宠物，而是执行一次“诞生生成”
- 初始化至少生成名字、初始气质、外观倾向、说话口吻、偏好倾向、对用户的第一印象
- 初始化输入必须同时包含工作结构和沟通风格

初始化推荐输入：

- 最近一段时间的任务分布与 Token 结构
- 近期沟通风格特征摘要
- 用户一句愿望或偏好描述
- 宿主环境与项目上下文
- 固定 seed

V1 原则：

- 使用风格特征摘要，不长期直接喂原始对话
- 初始化是生成“它是谁”，不是随机抽一个模板

### 三、单一核心互动

必须成立：

- 用户只有一个核心动作：拍一下
- 拍击并非随时可用，而来自宠物先吃出来的互动能量
- 拍击至少分轻拍、中拍、重拍三个层级
- 宠物会明确表达自己现在更想要哪一种拍法
- 拍对、拍偏、错过，反馈不同

V1 的关键不是“动作多”，而是“这一种动作足够深”。

### 四、关系性成长

必须成立：

- 它会记住你更常怎么回应它
- 它会因关系形成不同的称号、口吻、小习惯和外观倾向
- 不是固定进化树，而是在大阶段内持续分化
- 每个人养出来的都应该不同

V1 的独特性不靠“皮肤数量”，而靠“同一骨架下长出不同个体”。

### 五、AI 的角色

V1 中 AI 的角色固定为：

- 宠物灵魂生成器
- 外观倾向生成器
- 阶段性自我解释生成器

V1 中 AI 不直接负责：

- 实时数值生长
- 每一轮互动的底层规则判断
- 无约束的自由美术生成

分工必须稳定：

- 规则系统负责“它怎么活”
- AI 负责“它是谁”

## V1 非目标

V1 不做这些：

- 多宿主全部接通
- 多动作交互面板
- 完整图鉴宇宙和复杂成就系统
- 完整商业化层、排行、社交分享
- 每次互动都调用 AI 生成内容
- 完全自由的图像生成宠物体系

这些不是不做，而是不能抢走 V1 对“生命感、独特性、资源感知”的优先级。

## V1 成功标准

### 用户体验标准

- 用户第一次看到时，会觉得“这只不是系统默认发的一只”
- 用户能看懂 Token 怎么转成粮食，粮食怎么转成生命反馈
- 用户会开始关心它“现在想要什么”
- 用户会因为拍对或拍偏，感觉自己是在和一个生命体相处
- 用户连续使用几轮后，会觉得它的性格和样子正在变得更像“自己养出来的”

### 系统标准

- 桌宠能稳定启动并单实例运行
- Codex Token 数据读取稳定
- 转粮、吃粮、积累互动能、拍击反馈主链路稳定
- 宠物状态持久化稳定
- 初始化生成的结果能长期保持一致，不会频繁跳变

### 产品标准

- 它已经不是统计面板，而是有关系感的桌面存在
- 它已经把“珍惜 Token 和算力资源”翻译成温柔而持续的生命反馈
- 它已经具备继续向 V2 扩展的骨架，但 V1 本身已经自洽

## V1 工作分解

### 工作流一：稳定主链路

目标：

- 确保 `Token -> 粮食 -> 吃 -> 能量 -> 拍 -> 成长` 全链路稳定

优先任务：

- 启动与单实例稳定
- 状态持久化稳定
- 转粮与自动吃粮边界稳定
- 主面板信息密度与可读性稳定

### 工作流二：完成初始化

目标：

- 让宠物第一次诞生就“成立”

优先任务：

- 设计初始化输入结构
- 设计沟通风格摘要器
- 定义灵魂与外观生成输出格式
- 补第一次诞生流程与展示

### 工作流三：做深单一互动

目标：

- 让“拍一下”真正有关系深度

优先任务：

- 继续强化期待、错过、拍偏、拍准反馈
- 做出更明显的动作与表情差异
- 让不同宠物形成不同的等待与回应方式

### 工作流四：做出独特成长

目标：

- 让用户感觉“这是我养出来的这只”

优先任务：

- 关系历史更深入参与称号和习惯生成
- 外观倾向从静态标签走向更明显的身体差异
- 阶段性自我解释更像一个生命体的叙事

## V1 当前已具备基础

当前已经有的能力：

- 本地桌面桌宠可运行
- Codex 真实 Token 已接通
- 外部标准事件源已预留接口
- Token 已可转粮并入粮仓
- 宠物已会自动吃粮、积累亲和能与躁动能
- 交互已收敛到“拍一下”
- 已有拍准、拍偏、错过、等待反馈的雏形
- 已有长期关系影响称号、习惯和特征的基础
- 设计原则和 V1 目标已明确

这意味着：

- V1 已经不是从零开始定义
- 接下来重点不是再发散，而是把初始化与生命感最后补实

具体取舍判断见：`docs/v1-design-checklist.md`
发布前逐项验收见：`docs/v1-acceptance-checklist.md`

## V1 下一阶段优先级

接下来按这个顺序推进：

1. 先完成宠物初始化方案
2. 再把初始化接进首次诞生流程
3. 再继续加强等待、拍准、拍偏、被忽略时的动作差异
4. 最后再补 AI 阶段性自我解释

## 版本边界判断标准

以后如果有新想法，统一用这三个问题判断要不要进 V1：

1. 它是否直接增强“这只宠物是活的”这件事？
2. 它是否直接增强“这只宠物是我养出来的”这件事？
3. 它是否直接增强“Token 背后是珍贵资源”这件事？

如果三者都不是，原则上不进 V1。
