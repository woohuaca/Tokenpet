# 吞金兽 Token Pet 下一阶段精进方向

## 结论

吞金兽 Token Pet 下一版应把重点从“继续解释得更清楚”切到“让宠物更好玩、更想看、更想互动”。

不是放弃主循环，而是在主循环已经初步成立后，优先把用户最直接感受到的两层补强：

- 形象设计要更有辨识度
- 互动反馈要更有动态爽感

底层逻辑仍然围绕这条闭环：

`Token 花费 -> 完整记录 -> 分类解释 -> 转粮进食 -> 代谢平衡 -> 生长养成 -> 情绪回应 -> 关系沉淀`

下一阶段建议并进五个方向：

1. 完整 Token 账本
2. 生长养成可信度
3. 形象设计与互动表现力
4. 调试与验证效率
5. 公开分发体验

一句话说：先让“它长得更有记忆点、动得更有情绪、互动时更有爽感”，同时继续稳住底层养成逻辑。

当前判断再往前推一版，最需要做的不是新玩法，而是：

## 下一版本最该做什么

### 结论

下一版本最该做的是 `宠物形象设计与互动动态增强`。

它要解决的不是“逻辑有没有”，而是“用户看到它的第一眼，有没有喜欢；和它互动那一下，有没有感觉”。

如果这一层不补，吞金兽虽然已经有：

- 完整 Token 明细账
- 今日 / 本周汇总
- 调试入口
- 基础状态差异
- 第一版解释能力

但用户看到的仍然更像：

- 有数据
- 有动画
- 有反应

却还不够像：

- 这是一只我真的想养、想拍、想多看一眼的小兽

### 为什么它现在是最高优先级

因为当前最大的断点，已经不是“记不到”或“解释不了”，而是“宠物的存在感和好玩感还不够强”。

已经补上的能力：

- Token 长期记录
- 汇总统计
- 快速回归关键状态
- 图鉴解释区

现在最缺的能力：

- 把不同长势、情绪、互动诉求做出一眼能分辨的视觉状态
- 把“等待 / 拍准 / 拍偏 / 错过 / 进食 / 胀仓 / 舒展”拉开成更有节奏的动态反馈
- 让用户在不打开图鉴时，也能从它的外形和动作里感到差异

这一层如果不补，产品会停留在“逻辑成立”，但还没有到“陪伴成立”。

### 这一版要做成什么样

建议下一版本先围绕两条线推进：

1. 形象设计更有辨识度
2. 互动反馈更有动态爽感

建议先落 5 个可感知改进：

- `长势外显`
  例如：舒展、节制、亢奋、压仓，都有不同的身体光环和体态重心。
- `互动诉求挂件`
  例如：想被轻拍时出现靠近型提示，想释放躁劲时出现更利落的诉求提示。
- `状态动作差异`
  例如：等待更克制，拍准后有余波，拍偏会卡顿，错过会回缩。
- `成长个体差异`
  例如：不同灵魂与性格种子在耳、尾、配件、轮廓上继续拉开。
- `低侵扰动态氛围`
  例如：不是一直乱动，而是在关键状态下出现一眼能感到“它现在不一样了”的动态层。

### 为什么不是先做别的

不是先扩新系统：

- 因为现在最大短板不是系统数量，而是宠物“活没活起来”。

不是先继续堆很多解释：

- 因为解释面板已经有第一版，再往下加，边际收益不如先把感知层补强。

不是先做更复杂的养成分支：

- 因为用户还没充分被形象和互动吸住，分支再多也不一定更有感觉。

所以，下一版本最优先的不是“扩深逻辑”，而是“强化感知层，让用户更想跟它发生关系”。

### 这一版完成后的价值

这版做完，吞金兽会从：

- 会记录 Token
- 会做反馈
- 会解释自己

往前走到：

- 会让人第一眼记住
- 会让人更想拍一下、再看一眼
- 会把“养成结果”直接显在外形和动态上

这是从“功能成立”走向“陪伴成立”的关键一步。

## 一、完整 Token 账本

目标：

- 所有 Token 花费都能被长期记录、追溯和解释
- 运行态缓存可以截断，但历史账本不能截断

已经完成：

- `runtime/token-spend-ledger.jsonl` 作为完整 append-only 明细账
- `runtime/token-spend-ledger-index.json` 防止重复写入
- bridge 启动时扫描历史 Codex session 做回填
- `runtime/token-spend-summary.json` 输出今日 / 本周 / 长期汇总
- 图鉴已经能读取完整账本汇总

下一步精进：

- 增加“今天吞了多少 Token，长出了什么变化”的解释入口
- 增加更直接的用户可读摘要，而不只是聚合数字
- 增加“哪类 Token 正在主导这只吞金兽”的解释

验收标准：

- 用户能回答“我的 Token 去哪了”
- 用户能看到“这些 Token 怎么变成宠物变化”
- 用户能长期回看，而不是只看到最近几条

## 二、生长养成可信度

目标：

- 宠物变化不能只是数值涨跌，而要像真实代谢
- 用户能理解“为什么它现在兴奋、压仓、细养、胀仓”

下一步精进：

- 把 `Token -> 粮食 -> 进食 -> 代谢 -> 长势` 的解释做成可视化摘要
- 让不同任务类型真正影响身体倾向、动作倾向和长期性格
- 加强“过量 Token / 垃圾 Token / 高质量 Token”的差异反馈
- 建立成长阶段的明确里程碑，而不是只靠数值变化

验收标准：

- 同样花 Token，不同任务结构会养出明显不同的吞金兽
- 用户能感到“这是我这段工作方式养出来的结果”

## 三、形象设计与互动表现力

目标：

- 让吞金兽不只是会回应，而是长得更鲜明、动得更有趣、互动时更有触感
- 重点不是更花，而是更好玩、更好认、更有状态差异

下一步精进：

- 把舒展 / 节制 / 亢奋 / 压仓做成一眼可辨的动态外显
- 把出生、等待、进食、拍准、拍偏、错过的动作差异继续拉开
- 增加互动诉求的轻提示挂件，让“它现在想要什么”更直观
- 继续强化个体差异，让不同吞金兽在轮廓、耳尾、配件上更容易被记住
- 保持低侵扰原则，只在关键状态增强动效，不做持续打扰式热闹

验收标准：

- 用户一眼能分出“它今天是舒展、压仓、还是在等我”
- 用户互动时能明显感觉“拍准”和“拍偏”不是一回事
- 用户不是只觉得它可爱，而是觉得“这只真的有自己的状态和脾气”

## 四、调试与验证效率

目标：

- 不依赖真实 Token 等待，也能快速验证状态表现
- 每个关键状态都能一键触发和回归

下一步精进：

- 把本地调试入口继续做成稳定回归工具：
  `出生 / 等待 / 进食 / 拍准 / 拍偏 / 错过 / 认主`
- 增加状态快照导出
- 增加账本统计检查脚本
- 增加视觉状态验证清单

验收标准：

- 开发时 5 分钟内能看完 7 个关键状态
- 改动作和文案时不需要等真实 Codex 事件自然触发

## 五、公开分发体验

目标：

- 陌生用户下载后能看懂、装上、启动、反馈
- 产品名字、入口、安装说明和问题反馈统一成“吞金兽 Token Pet”

下一步精进：

- 更新 Release 文案与截图
- 准备“给朋友推荐”的短介绍
- 增加首次安装引导
- 增加常见问题：看不到、闪烁、权限、Token 不增长

验收标准：

- 用户不用问作者，也能完成首次启动
- 推荐给别人时，一句话能讲清楚它是什么
