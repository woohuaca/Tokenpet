# 吞金兽 Token Pet 下一阶段精进方向

## 结论

吞金兽 Token Pet 接下来不应该先扩很多新玩法，而应该把已经成立的主循环继续做深：

`Token 花费 -> 完整记录 -> 分类解释 -> 转粮进食 -> 代谢平衡 -> 生长养成 -> 情绪回应 -> 关系沉淀`

下一阶段建议并进五个方向：

1. 完整 Token 账本
2. 生长养成可信度
3. 情绪价值密度
4. 调试与验证效率
5. 公开分发体验

一句话说：先让“它为什么这样长、为什么这样回应我”变得更可信，再继续扩功能。

当前判断再往前推一版，最需要做的不是新玩法，而是：

## 下一版本最该做什么

### 结论

下一版本最该做的是 `生长养成解释面板`。

它要解决的不是“能不能继续长”，而是“用户能不能看懂它为什么这样长”。

如果没有这一层，吞金兽虽然已经有：

- 完整 Token 明细账
- 今日 / 本周汇总
- 调试入口
- 基础状态差异

但用户看到的仍然更像：

- 有数据
- 有动画
- 有反应

却还不够像：

- 这是我这段工作方式真正养出来的结果

### 为什么它现在是最高优先级

因为当前最大的断点，已经不是“记不到”或“验不了”，而是“解释还没真正到达用户”。

已经补上的能力：

- Token 长期记录
- 汇总统计
- 快速回归关键状态

现在最缺的能力：

- 把 `今天 / 本周的 Token 结构`
- 翻译成 `它为什么兴奋 / 压仓 / 细养 / 胀仓`
- 再翻译成 `它为什么现在想靠近你、等你、吃粮、或者闹别扭`

这一层如果不补，产品会停留在“后台逻辑成立”，但还没有到“用户心里也成立”。

### 这一版要做成什么样

建议下一版本先做一个轻量但够清楚的解释面板，直接回答四个问题：

1. 今天它主要吃了什么
2. 这些 Token 让它长出了什么状态
3. 为什么它现在是这个情绪 / 长势
4. 你接下来怎么回应，会更适合它

建议面板固定输出五块内容：

- `今日吞食摘要`
  例如：今天主要吃的是开发粮，量高、质量偏稳。
- `代谢结果`
  例如：这一轮更多转成了活力和长势，也堆出了一些仓压。
- `当前长势判断`
  例如：它现在偏“猛长型” / “胀仓型” / “细养型”。
- `当前情绪原因`
  例如：它现在等你，不是随机撒娇，而是互动能量已积够、但仓里还有没消化完的粮。
- `下一步建议`
  例如：适合轻拍、适合等它自己消化、适合补一次高质量任务粮。

### 为什么不是先做别的

不是先做更多动作：

- 因为动作差异已经有第一版了，再加动作，收益会开始递减。

不是先做更多情绪文案：

- 因为没有解释底座，文案越多越像“会说话的状态栏”。

不是先做更多成长分支：

- 因为用户还没真正理解现有成长是怎么来的，分支越多越难懂。

所以，下一版本最优先的不是“再丰富”，而是“把已成立的主循环翻译给用户看懂”。

### 这一版完成后的价值

这版做完，吞金兽会从：

- 会记录 Token
- 会做反馈

往前走到：

- 会解释自己
- 会让用户理解自己的养成结果

这是从“功能成立”走向“产品成立”的关键一步。

## 一、完整 Token 账本

目标：

- 所有 Token 花费都能被长期记录、追溯和解释
- 运行态缓存可以截断，但历史账本不能截断

已经完成：

- `runtime/token-spend-ledger.jsonl` 作为完整 append-only 明细账
- `runtime/token-spend-ledger-index.json` 防止重复写入
- bridge 启动时扫描历史 Codex session 做回填
- `runtime/token-spend-summary.json` 输出今日 / 本周 / 长期汇总
- 图鉴已经能读取完整账本汇总

下一步精进：

- 增加“今天吞了多少 Token，长出了什么变化”的解释入口
- 增加更直接的用户可读摘要，而不只是聚合数字
- 增加“哪类 Token 正在主导这只吞金兽”的解释

验收标准：

- 用户能回答“我的 Token 去哪了”
- 用户能看到“这些 Token 怎么变成宠物变化”
- 用户能长期回看，而不是只看到最近几条

## 二、生长养成可信度

目标：

- 宠物变化不能只是数值涨跌，而要像真实代谢
- 用户能理解“为什么它现在兴奋、压仓、细养、胀仓”

下一步精进：

- 把 `Token -> 粮食 -> 进食 -> 代谢 -> 长势` 的解释做成可视化摘要
- 让不同任务类型真正影响身体倾向、动作倾向和长期性格
- 加强“过量 Token / 垃圾 Token / 高质量 Token”的差异反馈
- 建立成长阶段的明确里程碑，而不是只靠数值变化

验收标准：

- 同样花 Token，不同任务结构会养出明显不同的吞金兽
- 用户能感到“这是我这段工作方式养出来的结果”

## 三、情绪价值密度

目标：

- 让吞金兽不只是记录 Token，而是持续给用户情绪反馈
- 重点不是更吵，而是更会回应

下一步精进：

- 把出生、等待、进食、拍准、拍偏、错过的动作差异继续拉开
- 增加“被记住感”：它能记住用户常见互动节奏
- 增加“陪伴感”：空闲时不打扰，但有轻微存在感
- 增加“关系感”：同一只吞金兽会因为长期互动长出偏好

验收标准：

- 用户不是只觉得它可爱，而是觉得“它有点懂我”
- 用户愿意每天看一眼、拍一下、观察它今天的状态

## 四、调试与验证效率

目标：

- 不依赖真实 Token 等待，也能快速验证状态表现
- 每个关键状态都能一键触发和回归

下一步精进：

- 把本地调试入口继续做成稳定回归工具：
  `出生 / 等待 / 进食 / 拍准 / 拍偏 / 错过 / 认主`
- 增加状态快照导出
- 增加账本统计检查脚本
- 增加视觉状态验证清单

验收标准：

- 开发时 5 分钟内能看完 7 个关键状态
- 改动作和文案时不需要等真实 Codex 事件自然触发

## 五、公开分发体验

目标：

- 陌生用户下载后能看懂、装上、启动、反馈
- 产品名字、入口、安装说明和问题反馈统一成“吞金兽 Token Pet”

下一步精进：

- 更新 Release 文案与截图
- 准备“给朋友推荐”的短介绍
- 增加首次安装引导
- 增加常见问题：看不到、闪烁、权限、Token 不增长

验收标准：

- 用户不用问作者，也能完成首次启动
- 推荐给别人时，一句话能讲清楚它是什么
