# 阿在 Token Pet 宿主接入协议

目标回答的 Charter 问题：How

## 结论

要同时支持 Codex、CC、Cursor，最稳妥的方式不是先绑死某一个宿主 API，而是定义统一事件协议，让所有宿主都把数据翻译成同一类宠物事件。

## 一、统一输入事件

宿主层最终只需要向核心引擎提交四类事件：

### 1. token_usage

表示一次可被喂养的 Token 消耗。

建议字段：

```json
{
  "type": "token_usage",
  "taskType": "coding",
  "tokensSpent": 4200,
  "outputQuality": "solid",
  "timestamp": "2026-04-07T20:30:00+08:00",
  "sessionId": "session-123",
  "source": "codex"
}
```

### 2. time_decay

表示一段时间流逝，用于触发饱腹度、活力和清洁度衰减。

```json
{
  "type": "time_decay",
  "hours": 3
}
```

### 3. care_action

表示用户主动照料宠物。

```json
{
  "type": "care_action",
  "action": "clean"
}
```

可选值：

- `clean`
- `play`
- `sleep`

### 4. weekly_summary

用于生成周报、挑战结果和进化解释。

```json
{
  "type": "weekly_summary",
  "periodStart": "2026-04-01",
  "periodEnd": "2026-04-07"
}
```

## 二、任务类型标准化

不同宿主原始标签会不同，但进入核心引擎前应统一归一成以下 8 类：

- `coding`
- `writing`
- `research`
- `review`
- `planning`
- `meeting`
- `support`
- `idle`

这样做的原因是：

- MVP 阶段先保证高可信解释，而不是追求无限细分
- 进化路线和营养逻辑可以稳定复用
- 多宿主数据可以横向比较

## 三、宿主职责边界

### 宿主适配层负责

- 读取原始 Token 数据
- 推断任务类型
- 评估产出质量
- 把数据映射为统一事件
- 将状态结果回写 UI

### 核心引擎负责

- 计算营养结果
- 更新宠物状态
- 推导情绪标签
- 推导成长阶段
- 推导进化形态
- 输出周度解释

## 四、各宿主建议接法

### Codex

优先接入，因为插件形态最直接。

建议能力：

- 会话级 Token 使用量
- Prompt 或任务标签
- 侧边栏宠物面板
- 周报卡片

### 为什么 Codex 应该继续做第一平台

结论：

吞金兽不应该只把 Codex 当作“第一个能装插件的地方”，而应该把 Codex 当作最先要长透的产品生态。

原因不是单纯因为当前技术接得上，而是因为 Codex 已经天然具备三层非常适合吞金兽的土壤：

1. 它本身就是高频 AI 工作现场。
2. 它天然有角色化、技能化、工具化并存的产品语言。
3. 它允许“插件入口 + 对话入口 + 本地 companion”组成复合体验，而不是只能做一个静态面板。

也就是说，Codex 对吞金兽的价值，不只是宿主价值，而是世界观价值。

### 从当前 Codex 外显系统里能学到什么

基于当前可见的 Codex 体验，可以先稳定抽出 4 个对吞金兽最重要的启发：

#### 1. Codex 不是把能力做成菜单，而是做成角色系统

它让复杂能力不只以功能名出现，而是以可记忆、可区分、可形成偏好的角色入口出现。

这对吞金兽的启发是：

- 不要只做“状态列表”
- 要逐步做成“型谱系统”
- 让不同成长结果更像不同角色，而不是同一只宠物的数值波动

#### 2. Codex 的识别感来自“图腾化”而不是“大插画”

当前可见的识别体系更接近：

- 简洁图标
- 有家族感的形状语言
- 有分组感的色彩
- 一眼可认的角色目录

这对吞金兽的启发是：

- 后续不需要先上复杂插画体系
- 更应该优先做稳定图腾、固定器官、家族配色和一眼可认的轮廓

#### 3. Codex 允许同一能力出现在多个入口

对用户来说，一项能力可以同时存在于：

- 插件列表
- starter prompt
- 对话入口
- companion

这意味着吞金兽不能只把自己理解成“桌宠程序”，而应理解成：

- 一个 Codex-native 的生命反馈层

桌宠只是其中最强情绪价值的一层，不是全部。

#### 4. Codex 的产品语言是“skill / agent / workflow”并存

这意味着吞金兽如果只做“宠物会动”，会显得太孤立。

更有潜力的方向是让它逐步长成：

- 能感知 workflow
- 能承接不同 skill 使用习惯
- 能把工作流结构翻译成角色变化

### 对吞金兽的产品判断

基于上面这些观察，吞金兽在 Codex 上最值得坚持的，不是“尽快多宿主”，而是：

- 先成为一个真正 `Codex-native` 的 Skill
- 再成为一个真正 `Codex-native` 的角色系统

这两件事顺序不能反。

如果先急着多宿主扩张，会有两个问题：

1. 产品语言会变成泛化插件语言，失去 Codex 内的辨识度。
2. 还没长出自己的平台特征，就会被稀释成“任何 AI IDE 都能装的桌宠”。

### 吞金兽在 Codex 上最值得长透的 5 个方向

#### 1. Codex-native 入口体系

要让吞金兽不只是“安装后能启动”，而是形成稳定入口矩阵：

- starter prompt
- 插件卡片
- 图鉴入口
- 桌宠 companion
- 调试/形态切换入口

目标是让用户无论从插件、对话还是桌面进入，都觉得是同一只吞金兽。

#### 2. Codex-native 角色谱系

要把成长结果继续从状态，升级成角色分支。

建议后续固定往三层做：

- 主系：仓压系 / 猛长系 / 贴贴系
- 图腾：每系一个固定识别符号
- 电子器官：每系 2-3 个稳定器官语言

目标不是增加组合数，而是增加记忆度。

#### 3. Codex-native 工作流映射

吞金兽的独特性应该来自：

- 你在 Codex 里做了什么类型的工作
- 你用哪些 skill
- 你是偏实现、偏研究、偏评审、偏支持

它后续最值得做的，不是更多随机情绪，而是更稳定地把这些工作流差异翻译成：

- 饮食结构差异
- 器官差异
- 表情差异
- 长势差异

#### 4. Codex-native 图鉴宇宙

图鉴不应只是状态面板。

在 Codex 语境里，图鉴更适合逐步长成：

- 角色档案
- 型谱目录
- 成长记录
- 工作流痕迹收藏页

也就是说，图鉴是“我在 Codex 里养出了什么”，而不是“我现在还有多少饱腹度”。

#### 5. Codex-native 社交推荐性

Codex 上一个很重要的传播方式，不只是发布，而是“看见别人的工作台很想自己也装一个”。

所以吞金兽要尽量增强：

- 截图传播性
- 一眼认出是 Codex 生态里的东西
- 一眼认出这只宠物和别人的不一样

### 当前阶段最该做的取舍

结论：

如果目标是“最大化在 Codex 上成长起来”，当前最优先的，不是再扩平台，而是先补下面三件事：

1. 角色谱系成立
2. 图腾与电子器官成立
3. Codex 入口的一致体验成立

只有这三件事成立，吞金兽才会在 Codex 里从“一个插件”长成“一个被记住的角色系统”。

### Cursor

重点观察编辑器侧 UI 与 Webview 面板。

建议能力：

- Chat Token 使用量
- 当前工作区上下文
- 命令触发照料动作

### CC

建议先走轻量模式，只做会话后总结和周报，不抢第一版实时体验。

建议能力：

- 会话 Token 汇总
- 会话类型标签
- 面板化状态展示

## 五、MVP 接入原则

第一版必须坚持以下原则：

- 自动喂养优先，减少手动输入
- 分类准确优先于分类精细
- 解释可信优先于动画复杂
- 先做单宠物，再考虑多宠物或宠物社交

## 六、后续扩展

当 MVP 验证通过后，可以继续增加：

- 宠物图鉴
- 不同人格初始蛋
- 团队宠物
- 周挑战和成就系统
- Token 预算模式
