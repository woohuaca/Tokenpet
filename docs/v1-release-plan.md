# 阿在 Token Pet V1 发布方案

## 发布结论

V1 当前不建议一开始就作为“多平台公开插件”同时铺开，而应采用：

- Codex-first
- 本地桌面 companion 优先
- 在 Codex 范围内转入正式发布

一句话定义：

- V1 发布 = 一次以 Codex 为首发宿主、以本地桌宠 companion 为主体体验、并在 `Codex + macOS` 范围内按正式版口径发布

## 为什么这样发布

当前产品真正成立的体验，不只是“插件装上去”，而是：

- 真实 Token 转粮
- 桌面宠物持续存在
- 首次诞生初始化
- 单一互动“拍一下”
- 长期关系沉淀

这意味着 V1 更接近：

- 插件入口
- 本地 companion 主体
- 规则系统与身份生成协同运行

而不是一个单纯的编辑器内插件卡片。

因此，V1 的发布策略应优先保护完整体验，而不是优先追求渠道覆盖。

## V1 首发位置

### 第一发布面：Codex 首发正式版

形式：

- 本地插件目录
- 桌面桌宠 companion
- 启动脚本
- GitHub 仓库与 release 包

原因：

- 当前这条链最完整
- 真实 Token 来源已经接通
- 桌宠体验已经跑起来
- companion 形态更符合产品本质

V1 应先把这条链按正式版标准跑稳，而不是先进入多平台分发复杂度。

### 第二发布面：Cursor 公测渠道

建议时机：

- V1 核心链稳定后
- 至少完成一轮 Codex 私测验证后

原因：

- Cursor 更适合做后续公开 beta 扩散
- 但不适合作为当前第一发布面

### 第三发布面：Claude Code 插件包

建议时机：

- 当桌宠主体和插件入口已经解耦更稳定之后

原因：

- 适合技术用户
- 但不适合当前作为首发体验容器

## V1 发布形式

V1 当前在 Codex 首发面建议采用正式版本号：

- `v1.0.0`

发布包建议包含：

1. Codex 本地插件目录
2. 桌面桌宠 companion 可执行文件
3. 启动脚本
4. 自启安装与卸载脚本
5. 最小安装说明
6. 最小使用说明
7. 已知限制说明
8. 隐私与本地数据说明

更具体的历史 `dev preview` 包内容清单见：`docs/v1-dev-preview-package.md`
历史 `dev preview` 产物命名与目录规范见：`docs/v1-dev-preview-artifact-spec.md`
当前 Codex 正式发布说明见：`docs/v1-codex-release-notes.md`

## V1 发布节奏

### 阶段一：内部收口

目标：

- 把 V1 体验与稳定性补到可私测

完成标志：

- 初始化已经成立
- 主循环稳定
- 状态不丢
- 单实例稳定

### 阶段二：Codex 正式首发

目标：

- 在 Codex 范围内以正式版口径落地

当前首轮内部验证记录见：`docs/v1-validation-2026-04-08-round-01.md`
启动与单实例专项验证见：`docs/v1-validation-2026-04-08-round-02.md`
短周期重复启停回归见：`docs/v1-validation-2026-04-08-round-03.md`

当前首发用户：

- 重度 Codex 用户
- 愿意安装本地工具的人
- 能接受当前范围仍限定在 `Codex + macOS` 的用户

私测重点观察：

- 是否真的感到“它活着”
- 是否真的感到“它不是默认宠物”
- 是否愿意在工作间隙回头看它
- 是否会因为关系和期待机制而继续养下去

### 阶段三：公开扩展准备

目标：

- 在核心体验稳定后，准备第二渠道扩展

这阶段再考虑：

- Cursor 渠道
- 更明确的安装器
- 更标准化的隐私说明

## V1 发布门槛

V1 不以“做完多少功能”为门槛，而以“核心生命体验是否成立”为门槛。

逐项验收清单见：`docs/v1-acceptance-checklist.md`
验收记录模板见：`docs/v1-validation-template.md`

必须同时满足以下六条：

1. 首次启动会生成一只带名字、灵魂和出生描述的宠物。
2. Codex 真实 Token 能稳定转成粮食并被宠物消费。
3. 宠物会自然积累互动能，并明确表达当前期待。
4. 用户只有一个动作“拍一下”，但拍准、拍偏、错过有明显不同反馈。
5. 长期关系会沉淀成称号、习惯、外观倾向和状态差异。
6. 桌宠连续运行和重启后，不会多开、丢档、乱跳。

只要六条里有一条不稳，都不建议叫“正式发布”。

## V1 发布前还必须补齐的工作

### 一、首次诞生体验收口

还要补：

- 第一次见面的完整迎接感
- 更明确的“这是你的这只”确认
- 首次诞生后的引导话术
- 第一次轻拍完成认主时的明确完成反馈

### 二、发布级稳定性

还要补：

- 已通过当前专项复测：启动链稳定
- 已通过当前专项复测：单实例稳定
- 已通过当前专项复测：bridge 改由桌宠本体自愈拉起，PID 已能和真实进程对齐
- 已通过当前短周期回归：连续 3 轮启停后未出现双实例与 PID 偏移
- 还要补：一轮更贴近日常使用的长时观察
- 状态兼容与恢复
- bridge 异常恢复的长时观察

### 三、交互反馈可见性

还要补：

- 已完成：认主首拍已经有独立的完成演出和专属冲击反馈
- 已完成第一轮：拍准、拍偏、等待、失落、别扭已经有姿态和小特效区分
- 已完成第二轮：气泡条、边框、尾巴偏移和状态标签会跟着认主、拍准、拍偏、等拍变化
- 还要补：拍准、拍偏、错过之间的强弱动作还可以再拉开一档

### 四、最小发布配套

还要补：

- 已完成：README 已补最小安装、启动、关闭、重置说明
- 已完成：README 已补本地数据与隐私说明
- 已完成：已补面向 Codex 首发用户的短上手卡 `docs/v1-private-beta-quickstart.md`
- 已完成：README 已补更明确的“停止自启 / 重置 / 关闭”入口说明
- 已知限制

## V1 发布后不立即做的事

这些不应阻塞 V1：

- 多宿主全面接通
- 完整图鉴宇宙
- 多动作交互面板
- 社交传播层
- 商业化层
- 完全自由 AI 美术生成

## 发布判断标准

如果我们还在争论某个功能要不要塞进 V1，就用下面三个问题判断：

1. 它是否直接增强“这只是活的”？
2. 它是否直接增强“这只是我的”？
3. 它是否直接增强“资源感知”？

如果三者都不是，原则上不进 V1 发布范围。
