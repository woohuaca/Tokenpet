# 阿在 Token Pet 版本计划

## 版本结论

阿在 Token Pet 接下来的版本推进，不应按“功能越多越好”来排，而应按三个问题来排：

1. 生命感有没有先成立。
2. 长时陪伴有没有先稳定。
3. 分发范围有没有在不稀释体验的前提下再扩出去。

因此，版本路线建议固定为：

- `v1.0.x`：把首发正式版跑稳，不再让“能不能启动、会不会闪、会不会多开”抢走产品判断。
- `v1.1.x`：把“第一次见面”和“每天继续养”的体验补到更像一个生命体，而不是一个已经可用但还偏工程感的桌宠。
- `v1.2.x`：把“这只是我养出来的”做深，让关系沉淀开始稳定地产生差异。
- `v1.5.x`：把公开测试和更大范围分发所需的安装、恢复、反馈与观测链补齐。
- `v2.0.0`：再进入多宿主、多系统或公共插件目录层面的扩展。

一句话说，先把一条小而完整的生命体验做深，再把渠道做宽。

## 为什么这样排

当前产品最成立的价值，不是“又一个插件”，而是把真实 Token 消耗翻译成可持续陪伴的生命反馈。

这条价值要成立，必须先守住四个优先级：

1. 先守住主循环。
   `Token -> 粮食 -> 吃 -> 期待 -> 拍一下 -> 关系变化 -> 长成独特个体`
2. 先守住长时稳定。
   如果它经常丢、闪、乱跳、重开，用户不会把它当成生命体。
3. 先守住独特性。
   如果每只都像默认模板，产品会退化成“会动的统计挂件”。
4. 最后才扩宿主和渠道。
   分发过早扩张，只会把本来就还在收口的体验问题放大。

所以，这份版本计划主要回答的是“先做什么才不会把产品做散”。

## 版本规划原则

### 一、按产品风险排，不按开发兴趣排

优先解决最容易直接破坏产品成立条件的问题：

- 启动稳定性
- 单实例与持久化
- 期待与回应是否真的有关系感
- 独特性是否足够明显

### 二、按闭环升级，不按零件升级

每个小版本都必须让用户感到一个完整能力变强了，而不是只看到零散优化。

例如：

- `v1.1` 不是“多几个 UI 修复”
- `v1.1` 应该是“第一次诞生和继续养这件事明显更成立了”

### 三、按门槛发布，不按日历发布

版本是否进入下一阶段，不看日期是否到了，而看当前阶段的成功标准是否达到。

### 四、渠道扩张必须晚于体验收口

如果首发范围内的生命感和稳定性还没跑稳，不建议把问题带到更多宿主和更多系统。

## 当前基线

当前已经完成的基线是：

- `v1.0.0`

它的定义不是“功能做完”，而是：

- 已经按 `Codex + macOS` 首发正式版口径发布
- 真实 Token 转粮与桌宠 companion 主链路已经成立
- 发布、公开仓库、启动链和基础显示层已经具备正式使用条件

## 版本总表

| 版本 | 核心目标 | 用户能明显感到的变化 | 进入下一版前必须满足 |
| --- | --- | --- | --- |
| `v1.0.x` | 稳定首发链路 | 能稳定看到、稳定养、不会反复出故障 | 启动、显示、单实例、持久化问题不再频繁打断体验 |
| `v1.1.x` | 收口生命感 | 第一次见面更成立，等待与回应更像在相处 | 新用户能明显感到“它活着，而且是我的” |
| `v1.2.x` | 做深独特性 | 养几天后能明显看出“我的这只”和别人的不一样 | 独特性差异进入行为、关系和外观层，不只停留在文案 |
| `v1.5.x` | 准备公开测试 | 陌生用户可以独立安装、启动、反馈 | 下载、安装、反馈、恢复链可以自助完成 |
| `v2.0.0` | 进入扩展阶段 | 产品不再只依赖单宿主单系统成立 | 首发链已被验证，扩渠道不会稀释体验 |

## 版本路线

## `v1.0.x` 首发稳定带

### 版本目标

让当前正式首发版从“已经可发布”进入“可以放心连续使用”。

### 为什么先做这个

如果桌宠的存在感被启动失败、显示异常、状态不稳打断，用户不会继续走到关系层。

### 必须补齐

- 启动、自启、重启、睡眠唤醒后的稳定性继续观察
- 桌宠显示层、跨空间与回屏逻辑继续收口
- bridge 异常恢复与状态持久化继续观察
- 已知问题要能被 README 和排障说明快速解决

### 发布门槛

- 连续多轮启停、重启、睡眠唤醒后不出现双实例和明显状态错乱
- 用户不会频繁遇到“看不见桌宠”“窗口闪烁”“脚本成功但桌宠已死”
- 当前已知限制文档与排障说明足够自洽

### 不做什么

- 不新增大范围新玩法
- 不急着扩到更多宿主

## `v1.1.x` 生命感收口版

### 版本目标

把“它活着，而且这只就是我的”从概念成立，提升到第一次使用就能明显感到成立。

### 这一版要解决的核心问题

当前 `v1.0.0` 已经能跑，但第一次诞生、认主、等待和回应之间，还偏“机制成立”，不够“情感成立”。

### 必须补齐

- 首次诞生体验更完整
- 第一次认主完成反馈更明确
- 等待、拍准、拍偏、错过之间的姿态和情绪差距再拉开一档
- 气泡、状态文案和小特效更能表达“它现在想要什么”
- 首次用户上手说明更短、更像邀请，而不是工程手册

### 成功标准

- 新用户第一次看到时，会更容易说出“它不是默认模板”
- 新用户能直觉理解自己是在“回应一个期待”，而不是“点一个交互按钮”
- 首周反馈里，“像生命体”相关正向反馈明显高于“像小挂件”

### 不做什么

- 不增加很多新动作
- 不做完整图鉴和复杂养成面板

## `v1.2.x` 独特性深化版

### 版本目标

把“我在养一只宠物”推进为“我在养我自己的这只宠物”。

### 这一版要解决的核心问题

如果关系历史、回应偏好和成长分化没有越来越明显，产品的新鲜感会很快耗尽。

### 必须补齐

- 关系历史更深入参与称号、口吻和小习惯生成
- 不同宠物在等待方式、被拍后的恢复方式、情绪残留上更分化
- 外观倾向从标签差异走向更容易被看出的身体差异
- 阶段性自我解释更像个体叙事，而不是状态播报

### 成功标准

- 老用户能更明确地说出自己的宠物和别人不一样在哪里
- 连续养几天之后，用户能察觉“我过去怎么对它，会继续影响它”
- 独特性反馈不只停留在名字和文案，而开始进入行为与外观层

### 不做什么

- 不把 AI 放进每次实时互动
- 不引入过重的自由生成成本

## `v1.5.x` 公开测试准备版

### 版本目标

把产品从“愿意折腾的人可装可用”推进到“陌生用户可以下载、安装、反馈、恢复”。

### 这一版要解决的核心问题

当前正式版已经能发布，但还偏向开发者友好，不够大众测试友好。

### 必须补齐

- 更明确的安装器或更稳定的安装流程
- 重置、备份、恢复边界更清晰
- 更标准的日志、问题反馈与版本排障入口
- 公开下载页与 Release 说明更面向陌生用户
- 对外测试时需要的最小指标与反馈回收模板

### 成功标准

- 陌生用户不需要口头辅导，也能完成安装、启动和反馈
- 团队能更快区分“产品方向问题”和“安装环境问题”
- 公开测试不会因为基础运维成本过高而拖慢产品判断

### 渠道策略

这一版完成后，再评估：

- Cursor 公测渠道
- 更广义的公开分享
- 为未来公共插件目录开放做准备

## `v2.0.0` 扩展版

### 版本目标

在不牺牲生命感主骨架的前提下，把阿在 Token Pet 从单首发面产品，扩成可多面分发的产品。

### 进入 `v2.0.0` 的前提

- `Codex + macOS` 这一条链已经被反复验证
- 核心生命循环、稳定性和独特性都已经被用户证明成立
- 公开测试阶段的问题已主要从“能不能用”转向“往哪扩更值”

### 可以考虑纳入

- 多宿主接入
- 更多系统支持
- 更成熟的安装器与更新机制
- 公共插件目录或更正式的分发入口

### 仍要坚持的边界

- 不为了渠道覆盖削弱“真实 Token 喂养 + companion 存在感”这条主骨架
- 不把产品退化成纯插件面板

## 接下来三个迭代怎么排

为了避免版本路线只停留在标题层，建议直接按下面三个迭代推进。

### 迭代一：`v1.0.x` 稳定性清扫

目标：

- 把会直接破坏存在感的问题尽量清掉

这一轮最值得做的事：

- 长时稳定性观察与记录
- 自启、睡眠唤醒、重启场景回归
- README 补排障入口与常见问题
- 启动链、显示层、bridge 恢复继续收口

交付结果应该是：

- 一个更稳的 `v1.0.x`
- 一组可重复执行的验证记录
- 一份外部用户也看得懂的排障路径

### 迭代二：`v1.1.x` 生命感收口

目标：

- 把“机制成立”推进到“情感成立”

这一轮最值得做的事：

- 首次诞生演出
- 认主完成反馈
- 等待、拍准、拍偏、错过的差异再拉开
- 更像生命体的状态文案和轻提示

交付结果应该是：

- 一个更适合首次体验传播的版本
- 新用户第一次安装后更容易理解的核心感受

### 迭代三：`v1.2.x` 独特性深化

目标：

- 让留存理由从“它挺有意思”变成“我想继续养这只”

这一轮最值得做的事：

- 关系历史更深地进入生成逻辑
- 宠物个体差异更进入行为层
- 外观差异更可见
- 阶段性自我解释更像个体叙事

交付结果应该是：

- 更强的个体记忆点
- 更高的连续养成理由

## 版本号规则

为了避免以后每次发版都重新解释范围，建议固定以下规则：

- `patch`：稳定性、兼容性、显示层、脚本链路、文案修正，不改变产品边界。
- `minor`：让一个完整用户能力明显升级，例如首次诞生体验、独特性成长、公开测试准备。
- `major`：分发边界或产品形态发生变化，例如从单宿主首发进入多宿主扩展。

对应到当前路线：

- `v1.0.1` 到 `v1.0.x`：稳定性与首发收口
- `v1.1.x`：生命感收口
- `v1.2.x`：独特性深化
- `v1.5.x`：公开测试准备
- `v2.0.0`：扩展版

## 当前最建议马上排进来的工作

如果从今天开始排接下来两到三个迭代，建议按这个顺序：

1. 继续补 `v1.0.x` 的长时稳定性与排障文档。
2. 进入 `v1.1.x`，优先把第一次诞生、认主完成感和等待反馈做深。
3. 再进入 `v1.2.x`，把关系历史如何持续塑造“这只就是我的”做实。

同时有一条不变主线：

- 每个版本都必须继续优化 `Token 统计 -> 分类 -> 转化 -> 代谢与消耗 -> 平衡 -> 生长与养成 -> 情绪互动` 这条链，而不是只优化局部表现。

不建议现在立刻转去做：

- 多宿主铺开
- 大量新动作
- 完整图鉴宇宙
- 社交传播层
- 商业化层

## 版本裁剪规则

如果后面某个想法不知道该不该进当前版本，建议只问三个问题：

1. 它会不会直接增强“它活着”的感觉？
2. 它会不会直接增强“这是我的这只”的感觉？
3. 它会不会直接增强“陌生用户真的能装上并养起来”的成功率？

如果三个问题都不能明确回答“会”，原则上不进入当前优先版本。

## 当前判断

阿在 Token Pet 的版本计划不应该是“从小玩具慢慢加功能”，而应该是：

- 先证明它是生命化产品
- 再证明它是可长期陪伴的产品
- 最后再证明它是可规模分发的产品

只要这个顺序不乱，后面的版本扩展就会越来越顺；只要这个顺序一乱，产品很容易被渠道需求和功能冲动带偏。
