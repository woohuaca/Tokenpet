# 阿在 Token Pet V1 设计决策清单

## 这份清单是干什么的

这份清单不是功能列表，而是 `V1` 的设计闸门。

它要解决的问题是：

- 当我们想加一个新功能时，如何判断它该不该进 `V1`
- 当一个想法看起来“更丰富”时，如何判断它是不是在把宠物做散
- 当实现已经很多时，如何继续守住“它活着、它是我的、它提醒资源珍贵”这三个核心目标

因此，这份清单要服务的不是“做更多”，而是“做更准”。

## 使用方式

每当我们想加入一个新功能、新页面、新动作或一层新 AI 生成能力时，都先按下面顺序判断：

1. 先过四个判断问题
2. 再看它属于哪一层设计
3. 再决定它应进 `V1`、延后，还是直接拒绝

## 第一关：四个判断问题

先问这四个问题：

1. 这项设计是在增强生命逻辑，还是只是在增加操作项？
2. 这项设计是在增强关系感，还是只是在增加反馈噪音？
3. 这项设计是在增强独特生成，还是在把宠物做回固定模板？
4. 这项设计是在帮助用户感知资源代价，还是在把 Token 又变回抽象数字？

判断规则：

- 如果四个问题里，前两个都答不上来，原则上不进 `V1`
- 如果第三个和第四个都明显削弱，原则上直接拒绝

## 第二关：它属于哪一层

### 一、生命层

它是否直接作用于：

- `Token -> 粮食 -> 吃 -> 能量 -> 反馈`
- 饥饿、活力、脏乱、成长这些生命状态
- 真实工作流怎样被翻译成生命输入

这层优先级最高。

### 二、关系层

它是否直接作用于：

- 宠物如何期待你
- 你拍准、拍偏、错过之后它如何变化
- 它如何记住你的照顾方式

这层优先级仅次于生命层。

### 三、生成层

它是否直接作用于：

- 它为什么是这一只
- 它如何从用户风格和工作结构里长出来
- 它如何逐渐变成不同个体

这层在 `V1` 中重要，但必须受规则系统约束。

### 四、表达层

它是否直接作用于：

- 用户能否一眼看懂它现在在发生什么
- 它的状态是否更有生命感
- 它的关系反馈是否更直觉

这层是必要层，但不能反客为主。

### 五、外围层

它是否主要属于：

- 更复杂设置
- 更完整图鉴宇宙
- 分享、排行、社交
- 商业化机制
- 多宿主全面适配

这层默认不应优先进入 `V1`。

## 第三关：V1 进入条件

一个想法只有满足下面至少一条，才值得认真考虑放进 `V1`：

1. 它直接强化主循环：`Token -> 粮食 -> 吃 -> 能量 -> 拍 -> 成长`
2. 它直接强化“这只是活的”的感受
3. 它直接强化“这只是我的”的感受
4. 它直接强化“Token 背后是珍贵资源”的感受
5. 它直接降低私测用户的安装、启动、理解或持续使用门槛

如果五条里一条都不满足，原则上不进 `V1`。

## V1 必须优先做的东西

这些属于 `V1` 核心，优先级要始终高于新玩法：

### 一、主循环可信

- 真实 Token 能稳定转粮
- 粮食库存、自动吃粮、互动能源链条稳定
- 主循环不是演示逻辑，而是真实工作驱动

### 二、首次诞生成立

- 第一次出现不是默认宠物
- 会生成名字、灵魂、外观倾向和第一印象
- 认主第一下有明确完成感

### 三、单一互动足够深

- 只保留“拍一下”
- 但拍准、拍偏、错过、等待、失落都要明显不同
- 它会表达“现在更想怎样被拍”

### 四、关系会留下痕迹

- 它会记住你怎么对它
- 会在称号、习惯、期待方式和外观倾向上逐渐体现
- 每个人养出来的都应该不同

### 五、资源感知成立

- 用户能看懂 Token 怎样转成粮食
- 用户能看懂什么是高质量喂养，什么是低质量消耗
- 整个体验是温柔提醒，而不是审计和责备

## V1 可以延后的东西

这些可以做，但不应抢占当前优先级：

- 更多交互动作
- 更复杂设置中心
- 更大图鉴体系
- 成就、排行榜、分享
- 完整多宿主接通
- 完全自由的 AI 美术生成
- 每次互动都生成长文案

这些东西通常是“锦上添花”，不是“让生命成立”的第一条件。

## V1 应直接拒绝的方向

只要出现下面倾向，就应高度警惕：

### 一、把宠物做回数据面板

表现为：

- 强调统计而不是生命反馈
- 强调数字堆叠而不是粮食、状态和关系
- 用户看完只知道花了多少，不知道宠物怎么活

### 二、把宠物做回固定模板

表现为：

- 进化路径完全写死
- 不同用户养出来的差异只停留在皮肤颜色
- AI 只是换几句台词，没有真正参与“它是谁”

### 三、把交互做回工具操作

表现为：

- 动作越来越多，但关系越来越浅
- 每个按钮都像系统命令
- 用户像在管理系统，而不是在陪伴桌边小兽

### 四、把资源感做丢

表现为：

- Token 又退化成抽象积分
- “喂得多就是好”覆盖了“喂得对更重要”
- 用户感知不到资源成本和高质量使用的差别

## 当前阶段的默认决策

如果后续再遇到取舍不清的想法，`V1` 默认按下面规则执行：

1. 优先把一个动作做深，不优先把动作做多。
2. 优先把关系状态做清楚，不优先把面板做复杂。
3. 优先把独特成长做出来，不优先补固定内容库。
4. 优先把真实 Token 主链路跑稳，不优先急着接所有来源。
5. 优先把私测用户能装、能跑、能理解补齐，不优先做公开扩散包装。

## 一句话判断标准

如果一个设计让用户更容易说出下面三句话之一，它就更可能属于 `V1`：

- “它真的像活着。”
- “这只就是我养出来的。”
- “我开始更在意自己把 Token 花在什么上。”

如果一个设计不能帮助用户更容易说出这三句话，它大概率不该优先进入 `V1`。
