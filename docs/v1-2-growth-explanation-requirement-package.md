# 吞金兽 Token Pet `v1.2.x` 生长养成解释面板需求包

## 版本结论

`v1.2.x` 的首要目标，不是继续加新玩法，而是把已经成立的主循环翻译给用户看懂。

这一版要解决的核心问题只有一个：

- 用户能不能明确理解“这只吞金兽为什么现在是这样”

一句话说，`v1.2.x` 要把产品从：

- 会记录
- 会反应

推进到：

- 会解释自己
- 会让用户理解自己的养成结果

## 版本要回答的问题

`v1.2.x` 必须同时回答四个问题：

1. 今天它主要吃了什么 Token？
2. 这些 Token 主要转成了什么状态变化？
3. 为什么它现在是这种长势和情绪？
4. 用户下一步怎么回应，更适合当前这只吞金兽？

如果这四个问题答不清，`v1.2.x` 就还只是“多了一层数据显示”，不算真正完成。

## 为什么这一版现在必须做

当前已经补上的能力有：

- 完整 Token 明细账
- 今日 / 本周 / 长期 Token 汇总
- 关键状态调试入口
- 进食、等待、拍准、拍偏、错过的基础差异

当前最明显的断点是：

- 数据已经有了
- 动作已经有了
- 但用户还不够容易把这两者连起来理解

这会导致产品停留在：

- “它有很多后台逻辑”

而没有真正走到：

- “这是我这段工作方式养出来的一只吞金兽”

所以，`v1.2.x` 的优先级不是再扩内容，而是把解释层补齐。

## 版本边界

### 这一版必须做

- 做出一版轻量但清楚的生长养成解释面板
- 让账本汇总和宠物当前状态形成因果连接
- 让用户能读懂“当前长势判断”和“当前情绪原因”
- 给出下一步建议回应，而不是只播报状态
- 让解释输出保持稳定、短、准，不堆术语

### 这一版明确不做

- 不增加大量新动作
- 不做复杂图鉴叙事系统
- 不引入新的大型养成子系统
- 不把 AI 放进每一轮实时解释
- 不扩多宿主、多系统、多渠道

## 解释面板的产品定义

### 它是什么

它不是统计报表，也不是帮助中心。

它是吞金兽对自己当前状态的一段“可读自我解释”，负责把：

- Token 结构
- 转粮结果
- 代谢状态
- 长势判断
- 当前期待

串成一个用户能快速理解的闭环。

### 它不是什么

它不应该像：

- 数据看板
- 调试控制台
- 长篇系统说明

如果用户打开后感觉是在看后台，而不是在看“这只吞金兽今天怎么了”，这版就偏了。

## 模块一：今日吞食摘要

### 要解决的问题

用户现在能看到账本和转粮记录，但还不够容易一眼理解“今天它主要吃了什么”。

### 目标状态

用户打开面板后，第一眼就能知道：

- 今天吞了多少 Token
- 主要来自哪类工作
- 质量大致偏高、偏稳还是偏薄

### 这版必须补齐

- `今日 Token 总量`
- `今日主任务类型`
- `今日主质量结构`
- `今日主要粮食来源`

### 表达要求

- 用结论句，不先丢原始数字
- 数字保留，但只做支撑
- 一屏内看懂，不要像表格

### 验收标准

- 用户能复述“它今天主要吃的是开发粮 / 写作粮 / 研究粮”
- 用户不会觉得今天的摄入结构像随机拼出来

## 模块二：代谢结果

### 要解决的问题

用户知道它吃了什么，不等于知道这些 Token 主要变成了什么。

### 目标状态

用户能明确知道：

- 这一轮更偏向长势、活力、亲和，还是更偏向仓压、躁动、残渣

### 这版必须补齐

- `本轮主要转成了什么`
- `是否出现仓压 / 过饱 / 高代谢 / 残渣偏高`
- `为什么它现在更想吃 / 更想拍 / 更想自己消化`

### 表达要求

- 不直接抛很多状态字段名
- 用“代谢结果句”替代“属性播报句”
- 同时保留一条可追溯的底层依据

### 验收标准

- 用户能说出“它现在不是随机躁，而是这几轮吃得太猛”
- 用户能分辨“长得快”和“长得健康”不是一回事

## 模块三：当前长势判断

### 要解决的问题

现在系统内部已经有长势倾向，但用户还不够容易理解这些倾向意味着什么。

### 目标状态

用户能读懂它当前更偏：

- `细养型`
- `舒展型`
- `猛长型`
- `胀仓型`

并理解这不是标签装饰，而是近期工作结构和代谢平衡共同养出来的结果。

### 这版必须补齐

- 当前长势名
- 一句长势解释
- 一句近期形成原因
- 一句对之后行为的影响

### 表达要求

- 长势命名固定，不频繁变
- 解释短，不写成长论文
- 原因要指向最近几轮，而不是抽象空话

### 验收标准

- 用户能理解“为什么这几天它越来越像这种长势”
- 用户会开始主动观察自己的工作结构是不是在把它往某个方向养

## 模块四：当前情绪原因

### 要解决的问题

当前情绪反馈已经存在，但用户未必总能理解“它为什么现在在等 / 别扭 / 收回去 / 靠近你”。

### 目标状态

用户能把当前情绪看成有原因的结果，而不是动画表现。

### 这版必须补齐

- 当前主要情绪判断
- 当前请求或期待
- 当前情绪与近期摄入 / 互动之间的关系

### 表达要求

- 先说情绪，再说原因
- 语气像它在表达自己，不像系统播报
- 不把每个状态都解释得太机械

### 验收标准

- 用户能说出“它现在在等我，不是因为定时触发，而是因为互动能量积够了”
- 用户会更愿意基于它的状态做回应，而不是随机点一下

## 模块五：下一步建议

### 要解决的问题

即使用户理解了当前状态，如果不知道“接下来怎么回应更好”，解释就还没闭环。

### 目标状态

用户能得到一个很轻的下一步建议：

- 适合轻拍
- 适合先喂
- 适合让它自己消化
- 适合先继续做高质量任务

### 这版必须补齐

- 一条当前最推荐动作
- 一条不要做什么的轻提醒
- 一条“为什么是这个建议”的极短依据

### 表达要求

- 建议必须单一，不给多选题
- 不能像命令
- 不能像攻略

### 验收标准

- 用户能在看完面板后知道“我现在最适合怎么回应它”
- 用户不会觉得系统在教自己刷最优解

## 输出结构建议

建议这一版面板固定输出五段：

1. `今日吞食摘要`
2. `代谢结果`
3. `当前长势判断`
4. `当前情绪原因`
5. `下一步建议`

建议每段采用：

- 一句结论
- 一句原因

总长度控制在用户 10 秒内能读完的范围内。

## 数据与规则边界

### 规则系统负责

- 汇总真实 Token 结果
- 判断主任务类型与质量结构
- 判断近期代谢倾向
- 生成当前长势与建议动作

### AI 在这一版的角色

- 可以参与后续“阶段性自我解释”的润色
- 当前这一版不必依赖 AI 才能成立

原则：

- 先把解释骨架做稳定
- 再考虑让表达更有个体气质

## 最低实现要求

如果 `v1.2.x` 先做最小交付面，至少要完成：

- 图鉴或状态面板新增解释区
- 基于已有 `token-spend-summary.json` 输出今日摘要
- 基于已有长势 / 代谢状态输出解释句
- 基于当前请求与互动能量输出下一步建议

如果连这四项都没有，这版就还没有真正进入“解释层”。

## 版本成功标准

### 用户体验标准

- 用户能更容易理解“它为什么现在是这样”
- 用户能复述最近几轮工作是怎样影响这只吞金兽的
- 用户会开始按它的状态去回应，而不是随机点击

### 系统标准

- 解释输出和底层状态保持一致
- 不出现明显自相矛盾的解释
- 面板不会因为一两条数据波动而频繁跳变口径

### 产品标准

- 吞金兽不再只是“会把 Token 变成反馈”
- 吞金兽开始具备“把自己的成长翻译给用户”的能力

## 当前判断

`v1.2.x` 真正要推进的，不是更多内容，而是：

- 让解释层第一次成立
- 让用户第一次看懂“这只是我养出来的”

只要这一步做准，后面继续做个体关系感、长期差异和阶段性自我叙事，才会越来越像产品在变深，而不是功能在变多。
